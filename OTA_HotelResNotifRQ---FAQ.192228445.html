WrittenConfInst

Explanation

For integration partners to be able to send confirmation/Modification/cancellation letters to their clients, they require the information held within the <WrittenConfInst> section of the OTA_HotelResNotifRQ

The following letter types can be sent via OTA_HotelResNotifRQ based on the templates with various letters:

  • Offers
  • Confirmations
  • Cancellations

A protel user can select the letter type that wishes to be sent, and this will populate the "WrittenConfInst" section of the OTA_HotelResNotifRQ message with one of the following corresponding code:

  • OFR
  • CNF
  • CXL

Rule for deleting data from profile:

pAir: No profile data is deleted

We use protective approach to secure the data in the PMS. Thus - no deletion is applied on existing profile data UNLESS an OTA_ProfileModifyRQ for the profile is sent

e.g.: Email address exists in the PMS profile record. Inbound reservation is sent where

  • <Email/> is sent. As result in the PMS new empty line for email is created
  • <Email/> element is not sent a all. As result in the PMS nothing changes for the email
  • new <Email DefaultInd="true" EmailType="1">test111386@protel.de</Email> is sent. As result in the PMS new email address is added to the guest profile - but the old one still remains

PWS:

  • If element is provided but empty => Result: remove data from profile
  • If element is not provided at all => Result: the data is not touched from the profile

Add/change/remove profiles in a reservation - pAir

Adding/changing/removing profiles in a reservation is handled per their @Role 

e.g.:

(1) Reservation exists in the PMS with 1 guest profile A. This means the guest is both the booker and the arriving guest. Inbound Modification of the reservation is sent where only one guest profile is sent but it is a new guest profile B. As a result, the PMS will keep the old guest profile A as a Booker of the reservation and add the new guest profile B as an Arriving Guest

(2) Reservation exists in the PMS with 2 guest profiles: A for Booker and B for Arriving Guest. Inbound modification of the reservation is sent where only 1 guest profile is sent and its role is "ArrvingGuest" C. As a result, the existing profile A of the reservation is not changed. Old arriving guest B is replaced with the new arriving guest profile C


Inbound modification of the Reservation Comment using OTA_HotelResNotifRQ - pAir

IPA-906

With pAir release 21.28 it is now possible to add/modify the reservation comment on existing reservations using OTA_HotelResNotifRQ inbound. Since the modifications of a reservation can be done by different partners, each of them can have a separate column. If the attributes are personalized per their originator - then each of the partners can manage their own comment section only. In order to avoid the PMS deleting the partner comment from the reservation or vice versa, the PMS front end will make the partner comment as read-only on the PMS UI. This requires the external comments to be labelled with a trailing sign: "##############################"

Sample inbound Comment section with vendor details provided:

Comment with vendor identifiers
<Comments>
<Comment CommentOriginatorCode="iHotelier" GuestViewable="true" Name="TC2">
<Text TextFormat="PlainText">test case 2 - add new comment 1 line</Text>
</Comment>
</Comments>

As a result, the follow up OTA_HotelResNotifRQ APIs will look like this:

Comment from PMS where inbound modification exist
<Comments>
<Comment CommentOriginatorCode="PMS-Frontend" GuestViewable="true">
<Text TextFormat="PlainText">Comment added from the PMS UI internally</Text>
</Comment>
<Comment CommentOriginatorCode="iHotelier" GuestViewable="true" Name="TC2">
<Text TextFormat="PlainText">test case 2 - add new comment 1 line</Text>
</Comment>
</Comments>

Unique comment originator identifier - e.g. the partner service ID. The value is viewable in the PMS FO and carried forward in the API so it is advisable to have end-user value added readable meaning.
If this attribute is not used, then: the inbound comment will always overwrite the existing one (if the existing one was entered as well with no @Name). If the inbound comment is split into multiple <Comment> lines in one API - due to the overwriting - only the last <Comment> value will be taken and recorded in the PMS.

the value is labelled with "FROM"

Comment/@GuestViewable

0..1

Boolean

true/false depending on the comment - if it should be guest viewable or not. By default - the value is "true"

no impact

Comment/@Name

0..1

String

Unique comment identifier - e.g. the partner comment ID. Unlike the @CommentOriginatorCode, the @Name one is meant to provide uniqueness of the respective comment line. This way 1 partner can manage multiple Comment lines and edit them separately without affecting other comment lines. If the value is not used, then for the resp vendor, with every reservation modification holding comment, the external comment will be always appended to the existing ones in the PMS reservation. The more the vendors and their modification on the same reservation - the larger the comment section in the PMS UI. This can lead to comments with multiple repeated lines and extensively large and unreadable comments for the Customers. 
If this attribute is not used, then: the inbound comment will always overwrite the existing one (if the existing one was entered as well with no @Name). If the inbound comment is split into multiple <Comment> lines in one API - due to the overwriting - only the last <Comment> value will be taken and recorded in the PMS.

the value is labelled with "SUBJ"

If a partner does not want to have a comment appended to the reservation - they can as well not provide this <Comments> block in their API at all.

CD Proxy Handling

Every incoming OTA_HotelResNotifRQ must go through the CD-Proxy EP in order for the CC data to be tokenized and hidden. At the CD-proxy, the data validated is only on level "is the message a valid soap 1.2 xml". The only element changed at this EP is the <Guarantee> one.

The flow is:

Incoming OTA_HotelResNotifRQ → CD-Proxy EP → ESB Async EP → PMS

There is no separate logging for messages that arrive at the CD-Proxy. The possible error that partners can receive is "Tokenization failed". The reasons can be in general of 2 types:

  • the CC detail is not valid
  • the message does not conform to soap 1.2 xml (incl namespaces, schema, etc)

EP will depend on the hotel location:

  • Ireland server: https://pci.protel.net/cd-proxy-io/pci/1/io/reservations
  • Sydney server: https://pci-sydney.protel.net/cd-proxy-io/pci/1/io/reservations
