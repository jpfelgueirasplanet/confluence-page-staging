Which general workflows are possible with a Kiosk system ?

CheckIn / CheckOut workflow:

A reservation is already existing in the PMS and a guest wants to process the CheckIn/CheckOut.


WalkIn workflow:

A reservation is not existing in the PMS and the guest arrives at the hotel and wants to create his reservation via Kiosk.




2

Is there a general overview about the order of the api calls for the CheckIn / CheckOut workflow and the WalkIn workflow ?

CheckIn workflow:

  1. IO_SystemStatusRQ to protel
  2. OTA_ReadRQ to protel (via ResID, Name, DateRange)
  3. HTNG_HotelFolioRQ to protel
  4. HTNG_HotelCheckInNotifRQ to protel
  5. HTNG_ChargePostingRQ to protel


CheckOut workflow:

  1. IO_SystemStatusRQ
  2. OTA_ReadRQ to protel (via ResID or Name)
  3. HTNG_HotelFolioRQ to protel
  4. HTNG_ChargePostingRQ to protel (invoice balance =0)
  5. HTNG_HotelCheckOutNotifRQ to protel


WalkIn workflow:

  1. IO_SystemStatusRQ to protel
  2. OTA_HotelAvailRQ to protel in order to get the RoomAvailabilities
  3. OTA_HotelResNotifRQ to protel
  4. OTA_ReadRQ to protel (via ResID or Name)
  5. HTNG_HotelFolioRQ to protel
  6. HTNG_HotelCheckInNotifRQ to protel
  7. HTNG_ChargePostingRQ to protel

Is there a way to set a specific folio for the debitor type of payments without configuring this manually for each reservation ? 

Yes this can be handled via the following X-Setup:

defaultmethodofpayment = xxx

Info: xxx is the number of the method of payment. Please note that the values in an MPE also need to be entered in a manner so that the methods of payment are also visible in the respective MPE hotel. 

Info: xxx = select *from zahlart = zanr

4

How can we get the needed mapping vlaues such as RateCodes, RoomTypes, Method of Payments, etc. ?

The IO_SystemDataRQ is made for that purpose. As the Integration partner receives all needed data within the response of IO_SystemDataRS.

You need to provide at least one of the elements Accounting, Profiles, Loyalty, Rooms, Reservations, Events, Rates or taxes, otherwise the RS message will hold no data.
Please refer to the documentation: IO_SystemDataRQ/RS - Old

5

Is it possible to leave out the WalkIn functionality and to implement only the CheckIn/CheckOut functionality ?

Yes. The Integration partner can decide based up on his Workflow if he wants to implement only CheckIn/CheckOut functionality or both so WalkIn AND CheckIn/CheckOut.



6

How can we decide which room status should be able to be checked in ? 

This depends on the decision of the property. Properties have different behaviour for this. i.e. some properties want only rooms with status "Cleaned" to be checked in. Other properties want rooms with "Cleaned" and/or "Checked" status available for CheckIn.  

Once confirmed with the property the following xsetup handles which kind of combination should be configured:
RoomAssignement xSetup:

  1. If there is no xsetup present for pws | io_checkin_accepted_room_status | - 

then the C/I will be possible in the first clean/checked room. If no room is available with this status, then C/I will fail with error RS to vendor.

2.  If there is xsetup present pws | io_checkin_accepted_room_status | 1 

then the C/I is enabled only for clean rooms

3.  If there is xsetup present pws | io_checkin_accepted_room_status | 4  

then the C/I is enabled only for checked rooms
