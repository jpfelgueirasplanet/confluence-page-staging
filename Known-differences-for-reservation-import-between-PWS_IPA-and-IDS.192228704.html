Profile Import data processing

In PWS/IPA, customer/partner preferences are not possible. 

If customer is referenced with correct ID (per mapping_to_external table), the resp record in kunden is updated

If specific element/attribute is provided in the inbound OTA_HotelResNotifRQ and is filled in, then it will be updated in the PMS

If specific element/attribute is provided in the inbound OTA_HotelResNotifRQ and is empty, then that content will be removed from the profile record in the PMS (unless the value is mandatory - e.g. Surname for private profiles, Type="1")

If specific element/attribute is not provided in the inbound OTA_HotelResNotifRQ, then the PMS value (If existing) remains unchanged

Automatic reservation split / assignment of room IDs

In IPA/PWS if the hotel is configured to automatically assign a room ID - then this rule will apply during the reservation import as well.

Circular message flow

Circular message flows

Circular Message Flow

E.g.:

  1. A hotel has 2 installations for diff integration partners
  2. Both partners share similar msgs per direction - e.g. both have reservations 2-way
  3. P1 sends a reservation to the PMS

With IDS reservation import, both partners will receive an outbound Reservation message with the same status/data as it came from P1. For P1 this can cause endless fup circle of retriggers and errors

With PWS, there is a CausedBy element stored per transaction. Based on it, the ESB can know when to drop a reservation to the originator in order to avoid a circular msg flow
