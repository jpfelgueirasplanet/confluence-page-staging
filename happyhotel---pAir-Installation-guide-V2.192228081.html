One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


2

Testing Occupancy

(OTA_HotelInvCountNotifRQ)

  • Trigger the OTA_HotelInvCountNotifRQ manually for 365 days
  • Verify in Kibana that the outgoing messages OTA_HotelInvCountNotifRQ is answered with RS message from the Integration system and no error occurs

https://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/invCount/365?step=3


 For more detail on the list of triggers, please visit this page: /PIODOC/pAir - Triggering messages - API Calls

3

Testing Statistics

(IO_StatsNotifRQ)

  • Trigger the IO_StatsNotifRQ manually for 45 days in the past 
  • Trigger the IO_StatsNotifRQ manually for 365 days in the future 
  • Verify in Kibana that the outgoing messages IO_StatsNotifRQ are answered with RS message from the Integration system



http://connect.protel.net/baoBackline/resources/interfaces/io/trigger/stats?step=3&daysBack=45

http://connect.protel.net/baoBackline/resources/interfaces/io/trigger/stats?step=3&daysForth=365

 For more detail on the list of triggers, please visit this page: /PIODOC/pAir - Triggering messages - API Calls

4

Sending Rates and Restrictions to happyhotel

  • After the installation is done, start a full data Export and check in Kibana that outgoing OTA_HotelRatePlanNotifRQ and OTA_HotelAvailNotifRQ is answered by their corresponding RS messages and no error occurs.

5

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    for the last 5 years and ending with 31.12.2099
  • depending on the size of the site and the numbers of reservations in the individual sites system,  for the first 5 past years one bulk for each year and after that one for the remaining time and for a small site you can run one bulk for the complete period of time (Approximate definition of huge property: Number of reservation > 200 000)

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


6

Data Validation Report - DQE

  • after checking if the bulk successfully finished, provide the integration partner the report
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Office (Büro) of the front office in pAir.
  • in case the bulk is split into different parts, it is recommend to generate differend DQEs one for each (split) bulk.


 

Two way Installation

Examples

1

Test incoming Rate and Restriction update

(OTA_HotelRatePlanNotifRQ and OTA_HotelAvailNotifRQ)

  • the integration partner should ask the customer to create a Test rate code
  • after the Test rate code has been created, ask the integration partner to push some rates and create one restriction
  • Make sure that the rate which needs to be updated is set to yieldable. Ask in the channel for which day the rate has been pushed, check the date and send a screenshot from the Protel Daily Rates if the rate for the specific date is showing in the list. 
  • as soon as this has been tested successfully repeat the tests for incoming rate updates for a rate or rate group chosen by the customer


2

Update the protel SMP page with the information about the installed customer
