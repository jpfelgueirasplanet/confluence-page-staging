One way Installation (Outbound)

Examples

1

Create a test reservation in the future with a test profile (at least one year after the system date) and provide the reservation number to your integration partner

  • Verify in Kibana that outgoing OTA_ProfileCreateRQ is answered by OTA_ProfileCreateRS and no error occurs, and as well that OTA_HotelResNotifRQ outbound is answered with OTA_HotelResNotifRS and no error occurs. 
  • Modify the name or address in the test profile and check if the OTA_ProfileModifyRQ was triggered, and an OTA_ProfileModifyRS was sent by the remote system. 
  • Make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs.
  • If the integration partner wants to test the outgoing message HTNG_HotelFolioNotifRQ, then a test reservation needs to be created on the current business day of the hotel customer. In such a case, check in Kibana the HTNG_HotelFolioNotifRQ and the corresponding RS message.


2

Create SCS Jobs in the SMP

Details

  • OTA_HotelStatsNotifRQ starting with the next day from the installation date, at 00:01
    ReportCode: HTNG_CurrentDailyPropertyStatistics_v1 for 45 days, hourly, starting with the present day.
    Please trigger this message immediately after it was created.

    (as additional info in case the integration partner asks: the message OTA_HotelInvCountNotifRQ will be triggered automatically after an EOD is done by the customer in the PMS)


OTA_HotelStatsNotifRQ

3

Verify in Kibana that outgoing OTA_HotelStatsNotifRQ is answered by OTA_HotelStatsNotifRS from TheCapital. 

Check with the integration partner if the message was received by their end as well. Provide the CID for these two messages if necessary.


4

The OTA_HotelInvCountNotifRQ message is triggered after each EndOfDay. Verify in Kibana after the EOD is done if the outgoing message OTA_HotelInvCountNotifRQ is answered by OTA_HotelInvCountNotifRS from TheCapital.

  • Activate in the IDS the rate message to be exported. Please note that availability Values and Restriction values are not needed, so no tick there is necessary!

7

Sending Rates to TheCapital

  • After the IDS installation and configuration is done, start a full data Export and check in Kibana that outgoing OTA_HotelRatePlanNotifRQ is answered by OTA_HotelRatePlanNotifRS and no error occurs.

8

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ) for the last 2 years and ending with the next two years, starting with the 1 January and ending 31 of December
  • for profiles (OTA_ProfileCreateRQ) for the last 2 years and ending with the next two years, starting with the 1 January and ending 31 of December
  • for folios (HTNG_HotelFolioNotifRQ) for the last 1 year and ending with the next 1 year, starting with the 1 January and ending 31 of December
  • for stay revenue (OTA_HotelStayInfoNotifRQ) for the last 1 year and ending with the next 1 year, starting with the 1 January and ending 31 of December

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


 

Two way Installation (Inbound)

Examples

1

Create a test reservation in the future with a test profile (at least one year after the system date) and provide the reservation number and profile ID to your integration partner

  • Request the integration partner to change the address or contact details in the test profile and verify in Kibana and the Front Office if the OTA_ProfileModifyRQ was received by the PMS and correctly processed. Check if the PMS answered with OTA_ProfileCreateRS without any errors.
  • Provide the integration partner with the Salutation list of codes, that are needed for him to change the Salutation in the profile messages 
  • Request the integration partner to send an incoming message HTNG_HotelFolioRQ for the test reservation created, and check in Kibana the HTNG_HotelFolioRS for any errors

2

Test an incoming Reservation LookUp via ResID and/or DateRange incl. ResStatus
(OTA_ReadRQ) 

  • ask the integration partner to do a reservation lookup using the ResID
  • check if the ResRetrieveRS is getting sent back to them with success
  • ask the integration partner to do a reservation lookup using a DateRange
  • check if the ResRetrieveRS is getting sent back to them with success
  • ask the integration partner to do a reservation lookup using a DateRange incl. ResStatus "In-house"
  • check if the ResRetrieveRS is getting sent back to them with success


3

Test an incoming posting 
(HTNG_ChargePostingRQ)

  • provide the ResID of the test reservation
  • integration partner posts a payment on the folio (i.e. -20 Euro)
  • open the folio in the PMS and check if the amount was posted correctly
  • in order to void the posting, the integration partner needs to post a payment with "+20 Euro"



4

Cancel your test reservation and verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS without any errors.


5

Update the developer portal with the information about the installed customer
