General steps

Examples

1

Request authentication details from the Integration partner for the new installation and request them to send you the authentication details  on a private Slack message.

Please provide them the HotelID as well - internally for Protel this is the CloudID 

  • Authentication method using specific given Endpoints for each hotel customer.
    These endpoints must be provided by Hotel IT or by TAC directly.     It MUST be an https:// endpoint!
    (No User / no Password needed.)

2

SSL Certificate

If TAC does require an SSL Certificate at protel, please inform them again that this is nothing protel is dealing with. 
TAC installs their software on a local machine at the client´s hotel. As pAir is Cloud based they need to make that local service connectible. 
The technicians of the hotel must deal with that as it has nothing to do with protelAir. This is about firewall settings etc. 
I/O sends messages with an Amazon Root CA 1 certificate. The customer's machine needs to be set up to handle this CA.

Special configurations:

Set a default cash register no. for TAC charge postings

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up. 

In order to register the incoming PostPayments with a specific cash register number inpAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

  1. Define a default cash register inpAir

System data /POS/ Payment methods /POS


2. Add a config. setup that is used to define the default cash register for all IO-related postings in the configuration editor

  • Add CashRegister and default folder in Configuration/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box

5

  • provide the token corresponding to the Integration to the integration partner (ex. screenshot token starting with K225)  

 

One way Installation


1

Create a test reservation in the future (at least one year after the system date)

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 
  • always provide the reservation number to your integration partner, so that the reservation can afterwards be cancelled


2

Create and start bulk upload

  • for reservations (OTA_HotelResNotifRQ) for the last 60 days incl. the current date.

Note: No DQE necessary.


BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


 

Two way Installation

Examples

1

Test incoming charges and payments

HTNG_ChargePostingRQ

Scenario 1: Room charges (Basic functionality)

  • Create a test reservation on the client´s database. Date of arrival should be set to today. Please check in the reservation (reservation does not need to be checked in)
  • wait for the HTNG_ChargePostingRQ from TAC
  • Validate if the incoming ChargePosting is displayed correctly on the clients invoice, looking at payment and revenue numbers
  • Let TAC send in a cancellation afterwards and validate the values.

Note: TAC basically tests a charge + a payment in one request (example). 


Scenario 2: Walk In guests for Spa 

  • Please create a default guest profile called "TAC payments" and exchange the profileID with TAC. A corresponding passerby invoice is going to get created once TAC pushes revenue and payments against that profileID. 
  • A "Walkin" to the Spa area will book his items pay the bill at the TAC machine. 
  • To make sure that the revenue is flowing into pAir´s reports, the items will get charged to a passerby invoice of that default profile. All daily charges will get collected on that passerby invoice. Same behaviour as in POS systems for cash payments in pAir. 
  • In the end of the day the client needs to check the invoice out ( Same behaviour as in POS systems for cash payments in pAir. )

Please note:

  • The Walkin guest will get his receipt by the TAC machine after payment. This has been agreed with TAC. The passerby invoice would be not ordered.
  • For some clients TAC needs special created methods of payments and items depending on which scenarios the client wants to use via TAC. 
    It has been agreed with TAC to coordinate these mappings between TAC and the client as the requested mappings can vary.
