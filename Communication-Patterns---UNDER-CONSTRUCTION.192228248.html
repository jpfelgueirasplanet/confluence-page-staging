Communication Patterns - UNDER CONSTRUCTION

Communication patterns

The following diagrams demonstrate the typical data flow inbound to and outbound from Protel. For more details on requirements for each, see the specific details under Data Flows. We support asynchronous message flow by default; Synchronous message flow will only be used if protel chooses to use that pattern for a specific integration. The acknowledgment (HTTP 200) will only confirm that the message has been received and it should be sent to protel prior to the processing of the message. Please consider that the timeout for the ACK to be sent to protel is 120 seconds - If your acknowledgment is delayed or not sent at all, protel will try to send the message again. If there is still no ACK from the vendor's endpoint, the call will be dropped in protel.

For all inbound messages (for instance OTA_ReadRQ (see Pull Model)) we will provide you with an acknowledgment (HTTP 200) within 15000 ms. Please retry the message whenever you get no answer or an HTTP status that is not 200. 


Push Model

The push model is a method of delivery of data. In the push model, one system will “push” notifications to another system by sending an (example) OTA_Hotel*NotifRQ to inform the receiving party about changes done by the sender. The receiving system will then process the notification to reflect the changes made by the sender (Sync) and reply with an OTA_Hotel*NotifRS acknowledgment to the sender.

Example 1

Facilitating synchronous behavior the caller sends the request message and indicates that the response should be sent on the same connection by supplying the WSAddressing “anonymous” URI in the wsa:ReplyTo element. Note that if the anonymous URI is used and there is both a wsa:ReplyTo and wsa:FaultTo element, both wsa:ReplyTo and wsa:FaultTo elements MUST use the anonymous URI; i.e., it is not possible to use the anonymous URI for wsa:ReplyTo but not wsa:FaultTo, or vice versa

Example 2

Asynchronous behavior is accomplished by implementing three “normal” web service methods: one on the service side that gets called by the consumer to start the asynchronous process (the method should have the _SubmitRequest suffix), and two methods that are implemented by the consumer to receive the result (suffixed _SubmitResult) or a fault (_SubmitFault). The response/reply message in each message exchange (the _SubmitRequest message exchange and _SubmitResult/_SubmitFault message exchange) is treated like normal and is correlated using WS-Addressing. In the sample, a UUID was used for CorrelationID.

  1. The provider sends the message, initiates the asynchronous process, and gets back an “empty” SOAP message. (e.g. Hotel*NotifRQ Example 1)

  2. The consumer completes the asynchronous process, and invokes the _SubmitResult/Fault method on the provider (vgl. Hotel*NotifRS Example 2)


Pull Model

The pull model is the method of requesting information or a transaction by a requestor using a OTA_*RQ message which is to be served by the responder using the corresponding OTA_*RS message. Like the Push model it only requires two messages (request, response). The data flow of the messages is as follows: Using a OTA_*RQ message, the requestor (System A) will send a message to query another system (System B) for information to be retrieved (e.g. doing a search using given criteria). Using the OTA_*RS message, System B will then return data that meets the requested criteria or is created upon the request data.
