Services installed per 1-Click via Service Store by a hotel user need to be configured prior to publishing their availability.


To simplify this configuration process and to cut down on manual steps for setting a service up, an automatic configuration makes the most sense.

Please note:

1-Click is so far supported for 1-way integrations (protel → 3rd party) with protel Air only!

This page should clarify FAQ's coming up while reading the above-mentioned page and explains the flow step by step.

3 - Either OAuth takes place or Auto provisioning of property data to Integration partner

OAuth

The Installation Wizard Module "OAuth2" redirects the user to the authentication page associated to the Service that is about to be installed.
The user then grants access to that service for protel.I/O which will generate a token that is then stored in the installation record to be used for authenticating calls from the ESB to the service's endpoint.

To authenticate and authorize the access to the external application, protel uses the Authorization Code Grant mode described in RFC6749 Section 4.1.

For now, protel.I/O only supports OAuth2 with non-expiring Access Tokens (tokens without refresh tokens attached to them when retrieved).

After successful authorization, the SMP backend retrieves the authorization code at the return URL stated in the initial OAuth request.
It then needs to check the authorization (and especially its state parameter) against the state information stored in place. If StateID, User session, Installed service ID and hotel match, the backend will retrieve the access token from the remote server by issuing an Access Token Request.
Once this request has been completed successfully, the installation wizard is shown in the frontend which will then restore its state and context and writes the retrieved acces token into the context's environment variable.

If the authorization request fails for some reason or the Access Token Request does not complete successfully, the wizard's OAuth2 error view is shown to the user. In this view, the user is informed that an error occured and that the installation cannot continue.
The user then has the choice to go "Back" which restarts the OAuth2 process (while leading to the "Initial view") or to cancel the installation process completely.

4 - Sending of Integration details to ESB/property and Integration partner

The IOConfigurationNotifRQ will go out after the 3rd party sent back the response to the Auto Provisioning Message.

This message gets sent first time as soon as the BOB installation has been created by the Auto Provisioning message.
It will also be sent every time the configuration of the manifest (e.g. message types sent, endpoint) gets changed or the installation of a property gets updated (activated or deactivated e.g.).

5 - Sending initial bulk uploads required by the system

If configured in the service's manifest (see example - only a part of the manifest), one or more Bulk Transfer Jobs will automatically be created during the Installation phase. 

Manual intervention will be required to inform you about the status of the installation.

Re-activation of a service could also trigger bulk uploads - this is a configuration on our side.

Example BTS job definition inside the manifest with all settable values
{
"esb": {
"btsJobs": [
{
"trigger": [
"install",
"activate"
],
"messageType": "OTA_HotelResNotifRQ",
"pausePerMessage": 1,
"recordsPerMessage": 1,
"sourceService": "io.protel.air",
"targetService": "dummy.service.rmsdummy",
"step": "PREPARE_AND_EXPORT",
"dateFromOffset": -760,
"dateToOffset": 1827,
"prioValue": 0.6
}
]
}
}

6 - Status of the Installation / Bulk upload

When the bulk transfer(s) are done, you will be informed either by protel support or a developer.

Furthermore, in case of an error during one of the bulk transfers belonging to a pending installation, a protel user manually fixes the problem with the installation and finishes the bulk transfer or cancel the job and create a new one.
