Installation Steps


General steps

Examples

1

Hotelpartner uses a different Token for each IO installation, but the same Endpoint .


Request authentication details (only the Token is needed) from the integration partner for the new installation and request them to send you the authentication details on a private Slack message or encrypted file by email and inform you by phone about the password.

Please provide them the HotelID as well - internally for Protel this is the CloudID 

Authentication method using the same Endpoint which is set as default and a specific Token given by the third party system for each hotel customer.

2

General Preparation pAir/NG

https://confluence.protel.net/display/PIODOC/pAir+%7C+General+preparation

3

Bob installation: one for pAir (if not existing) and one for the Integration  

  • Service ID for the Hotelpartner V2 integration:
    -  ifc.hotelpartner_gmbh.hpymv2

  • If necessary, please deactivate any other older I/O active installation for Hotelpartner 
  • provide the token corresponding to the integration partner on private Slack message or encrypted file by email and inform them by phone about the password.



4

Please set for the ResTarget the value protelAir/onPrem, the other Target is set as default for io.protel.air

5

Ask the customer whether specific payments need to be created or if Hotelpartner should use existing ones. Please verify that the payment was configured properly.

6

A list of articles needed to post revenues during HTNG_ChargePostingRQ needs to be given to the integration partner with Item.Code.

7

Special configurations:

  • Set a default cash register no. for Hotelpartner charge postings


This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up. 

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

  1. Define a default cash register in pAir

System data / POS / Payment methods / POS

2. Add an xsetup that is used to define the default cash register for all IO-related postings 

  • Go to System Data - Xsetups (https://app.protel.net/settings/xsetups)
  • Interfaces - IO
  • Enter the Cash Register Number under Default IO Cash Register

    Help page: https://confluence.protel.net/x/sgDVC


Hint:

Please have in mind that the payment methods in pAir have to be linked to -both- PMS AND CASH - so pAir is capable to interlink the payments to the cashier system you have configured in the step above. In case you refuse to do so, all your payments will be routed by pAir to cashier zero (0) automatically.

One way Installation


1

1

Create a test reservation in the future (at least one year after the system date) and provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs
  • Verify in Kibana that the outgoing messages OTA_HotelInvCountNotifRQ is answered with RS message from the Integration system


2

2

Test outbound profile creation/modification

(OTA_ProfileCreateRQ/OTA_ProfileModifyRQ )

  • Modify the profile of the test reservation created during the previous OTA_HotelResNotifRQ
  • Create test examples of the individual, company profiles. Verify in Kibana that outgoing OTA_ProfileCreateRQ is answered by RS and no error occurs.
  • Modify the individual company profiles by changing name and address details. Verify in Kibana that the OTA_ProfileModifyRQ is answered by the a OTA_ProfileModifyRS and no errors are present. 
  • Cancel your test reservation afterwards.


3

3

Create and start bulk upload

  • for reservations (OTA_HotelResNotifRQ) for the past 2 and next 2 years from the installation date
  • for profiles (OTA_ProfileCreateRQ) for the past 2 and next 2 years from the installation date

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one.

Two way Installation


1

Make a screenshot from pAir System Data with the short names of the rate codes managed by Hotelpartner for the customer (ask Pascal Ploch in Slack which rates), and a screenshot of the reservation codes and send them to Hotelpartner. 

Important notice for the Best Western properties: make a screenshot from pAir System Data with the short names of the rate groups as well


Rate codes
 

Market code

Source of business

2

Test an incoming Reservation and Profile Creation/Modification
(OTA_HotelResNotifRQ, OTA_ProfileCreateRQ / OTA_ProfileModifyRQ)

  • Create a test reservation (arrival date needs to be current PMS date) preferably with 2 guests per room and the profile should hold a test email address
  • Verify in Kibana that the incoming OTA_HotelResNotifRQ and OTA_ProfileCreateRQ is answered by RS and no error occurs.
  • Verify in Kibana that the incoming OTA_ProfileModifyRQ is answered by OTA_ProfileModifyRS and contains no errors.
  • Check amendments are appearing in the profile in protel.


3

Test an incoming Profile LookUp 
(OTA_ReadRQ) 

  • ask the integration partner to do a profile lookup 
  • check if the OTA_ProfileReadRS is getting sent back to them with success

4

Test incoming Rate update

(OTA_HotelRatePlanNotifRQ)

  • the integration partner should ask the customer to create a Test rate code if possible
  • after the Test rate code has been created, ask the integration partner to push some rates
  • Make sure that the rate which needs to be updated is set to yieldable. Ask in the channel for which day the rate has been pushed, check the date and send a screenshot from the Protel Daily Rates if the rate for the specific date is showing in the list

Note
Hotelpartner usually uses ony one rate for the reservation import and rate updates, but can use multiple rates as well. 


5

Test incoming Restrictions 

(OTA_HotelAvailNotifRQ)

  • Make sure that the rate which needs to be updated is set to yieldable
  • Ask the Integration Part in the Slack channel of the integration to push a restriction for a specific day on the Test rate code and then check in the Price availability for the given date if the restriction has been set

    If the property is a Best Western hotel:
  • Make sure the BW setting is placed correctly (BWI export settings) like in the screenshot example and provide Hotelpartner with the short name for the RateGroup Level that the customer decided to have the restrictions set (usually this is Level A with the short name A) 

The following restrictions are supported by Hotelpartner:

  -  Close Out
  -  Close to Arrival
  -  Minimum Length of Stay by stay

6

After the rate and restriction test, ask the Integration partner to clarify with the customer to have the Test rate code invalidated (through expired validity) after the verification of the Restriction message is done


7

Data Validation Report - DQE

  • after checking if the bulk successfully finished, provide the integration partner the report with the Net values
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Office (Büro) of the front office in pAir.



8

Test an incoming Trace
 (IO_ResModifyNotifRQ)  

  • Ask the partner to send a trace inbound to the test reservation
  • Verify in Kibana that the incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors.
  • Check the trace in the test reservation


9

Test an incoming invoice detail request with Hotelpartner

(HTNG_HotelFolioRQ) 

  • Request Hotelpartner to send a FolioRQ message
  • Verify in Kibana that incoming HTNG_HotelFolioRQ is answered by HTNG_HotelFolioRS and contains no errors.


10

Test an incoming Payment posting
 (HTNG_ChargePostingRQ)

  • Request Hotelpartner to post a charge in the folio of the test reservation
  • Verify in Kibana that incoming HTNG_ChargePostingRQ is answered by HTNG_ChargePostingRS and contains no errors
  • Check in PMS if the amount was posted correctly


11

Void the payment & Cancel the test reservation


12

Update the protel SMP page with the information about the installed customer
