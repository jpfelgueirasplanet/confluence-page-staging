Alerting for inconsistent data in PWS tables

  • Initiative
  • Exemplary scenario that triggers the error behaviour
  • PWS alerting
  • DB Entries

Initiative

During DB dump and import using ASCII of pws tables, a fatal behavior was observed. The pws_jobs and pws_livejob tables had all their new entries with ID = "0". Due to the missing unique identifier which is a primary key of the DB tables, no messages could be triggered out of the PWS and the IO interface was not usable. The preventive action is to NOT use ASCII dump of PWS tables especially for pws_jobs/pws_livejob tables. General corrective action is to drop the tables and re-start the pws. At that time the pws will generate those tables anew and all entries will be processed correctly afterwards. However - to be able to diagnose the situation correctly and prevent the IO from being used before it is too late, there are extra alerting signals/logging in PWS 3.8.X and later.

Exemplary scenario that triggers the error behaviour


  1. Stop PWS
  2. Export DB tables pws_jobs / pws_livejob in ASCII
  3. Drop the 2 DB tables
  4. Import the 2 DB tables from the ASCII
  5. Change the PWS build to the branch one
  6. Create a new reservation
    Note! There can be other scenarios leading to this error. All functionalities will be affected - not only reservations!

DB Entries

  • The definition of the pws_jobs, pws_livejob tables will be wrong
  • Every new entry in the resp tables will be with ID = "0"
  • The PMS functionalities and tables are not affected
SQL query table definition
SELECT t.name AS tname,c.name AS cname,c.system_type_id,m.definition,c.max_length,c.is_identity, type_name(c.system_type_id) as typename, c.is_nullable, m.name FROM sys.columns c JOIN sys.tables t ON t.type='U' AND t.object_id=c.object_id
JOIN sys.schemas s ON s.schema_id=t.schema_id
LEFT JOIN sys.default_constraints m ON c.default_object_id=m.object_id WHERE t.name='pws_livejob';
