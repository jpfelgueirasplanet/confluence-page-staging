One way Installation


1

  • enable LiveMode and activate ToggleTrigger for required message types for 1-way connection

2

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


3

Testing Occupancy

(OTA_HotelInvCountNotifRQ)

  • Trigger the OTA_HotelInvCountNotifRQ for 1 year once on the server of the customer where the PWS service is running 
  • Verify in Kibana that the outgoing messages OTA_HotelInvCountNotifRQ is answered with RS message from the Integration system
  • for Occupancy the PWS will send the OTA_HotelInvCountNotifRQ automatically 3 times a day at 5,14 and 22 o'clock

pattern:
http://IP-address(PWS):Port(PWS)/ota/hotelinvcountnotif/{MPE_hotel}/send/{START:YYYY-MM-DD}/{END:YYYY-MM-DD}

e.g.: http://127.0.0.1:8046/ota/hotelinvcountnotif/1/send/2020-08-27/2021-12-26

 For more details on the URL triggers please visit this page: on prem -Triggering messages - API calls (PWS)

4

Testing Statistics

(IO_StatsNotifRQ)

  • Trigger the IO_StatsNotifRQ for 45 days in the past once on the server of the customer where the PWS service is running 
  • Trigger the IO_StatsNotifRQ for 365 days in the future once on the server of the customer where the PWS service is running 
  • Verify in Kibana that the outgoing messages IO_StatsNotifRQ are answered with RS message from the Integration system
  • for statistics the PWS will send both reports of the IO_StatsNotifRQ automatically at EOD

Pattern

http://127.0.0.1:8000/io/statsnotif/send/{mpe}/{ReportCode}/{Days}

5

Sending Rates to happyhotel

  • After the installation is done, start a full data Export and check in Kibana that outgoing OTA_HotelRatePlanNotifRQ and OTA_HotelAvailNotifRQ is answered by OTA_HotelRatePlanNotifRS and OTA_HotelAvailNotifRS respectively, and no error occurs.

6

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    for the last 5 years and ending with 31.12.2099
  • depending on the size of the site and the numbers of reservations in the individual sites system,  for the first 5 past years one bulk for each year and after that one for the remaining time and for a small site you can run one bulk for the complete period of time (Approximate definition of huge property: Number of reservation > 100 000)

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

Check for failed messages after the bulk upload: Manual Correction (Support-Tool Use Cases) 

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


7

Data Validation Report - DQE


  • after checking if the bulk successfully finished, provide the integration partner the report
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Add-Ons of the front office
  • in case the bulk is split into different parts, it is recommend to generate differend DQEs one for each (split) bulk.

Testing incoming Rate update

(OTA_HotelRatePlanNotifRQ)

  • the integration partner should ask the customer to create a Test rate code
  • after the Test rate code has been created, ask the integration partner to push some rates
  • Make sure that the rate which needs to be updated is set to yieldable. Ask in the channel for which day the rate has been pushed, check the date and send a screenshot from the Protel Daily Rates if the rate for the specific date is showing in the list. 

  • as soon as this has been tested successfully repeat the tests for incoming rate updates for a rate or rate group chosen by the customer
