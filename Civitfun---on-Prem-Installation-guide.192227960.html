Latest update
automatic room assignment during C/I with onPrem: with PWS version 2.20.4 this is what happens:

If there is no xsetup present for pws | io_checkin_accepted_room_status | - 

then the C/I will be possible in the first clean/checked room. If no room is available with this status, then C/I will fail with error RS to vendor

If there is xsetup present pws | io_checkin_accepted_room_status | 1 

then the C/I is enabled only for clean rooms

If there is xsetup present pws | io_checkin_accepted_room_status | 4  

then the C/I is enabled only for checked rooms

Note: If rooms with status: checked and cleaned should be available for CheckIn this Xsetup does not need to be set!

Errors regarding the HTNG_ChargePostingRS

  • The third-party would not be able to process the payment because it took more than 30s for the HTNG_ChargePostingRS to be sent out. In Kibana The HTNG_ChargePostingRS will include a success

To mitigate the problem the following configuration settings are suggested as the may reduce the comserver load

  1. Make sure PWS is connected to its own taskserver
  2. The new taskserver should have nothing else running against it, no ckits, no IDS
  3. IDS (all configured instances) should be pointed to its own taskserver and not the comserver 
  4. Taskserver should be set to reload system data cache every 60 minutes
    TaskServer_Setup
  5. Set this on taskserver.ini as that is where the pws is pointing
    [system]
    debugevtimerexecutiontimer=1
    thresholdevtimer=900
  6.  Make sure passerby account is configured and available
  7. Add the xsetup on global level
    ComServer | Whitelist | GetComputerName 
  8. If there is a timeout or error, check what the taskserver displays on the screen.

12

Errors regarding the HTNG_ChargePostingRQ

  • The third-party would not be able to send a payment via HTNG_ChargePostingRQ as the RQ drops an error

What was the issue:
When an Int. partner sends a payment to protel via HTNG_ChargePostingRQ the message failed with the following error:



Error: <Error ShortText="ReceiveFailure: The underlying connection was closed: An unexpected error occurred on a receive. HttpResponse is empty. " Code="321" Type="1">Error -1 (UnknownError): ReceiveFailure: The underlying connection was closed: An unexpected error occurred on a receive. HttpResponse is empty.


Related to this the CheckOut (inbound) failed and the invoice did not get generated.

The reason for this issue was a table in the default invoice form of the property.

Solution:
After taking out the table part from the invoice form (see screenshot) the incoming HTNG_ChargePostingRQ worked fine and the related CheckOut (inbound) including the invoice from generation was success.

2

  • Test an incoming Availability check
    (OTA_HotelAvailRQ)
  • please check if the OTA_HotelAvailRQ was responded successfully with an OTA_HotelAvailRS
  • Test an incoming Reservation Look Up (via ResID; via Arrival/DepartureDate and ResStatus)
    (OTA_ReadRQ)
  • Create a test reservation in PMS
  • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  • Test an incoming posting
    (HTNG_ChargePostingRQ)
  • provide the ResID of the test reservation
  • integ. partner posts an amount on the folio (i.e. 10 Euro)
  • open the folio in the PMS and check if the amount was posted correctly
  • Test an incoming Check In
    (HTNG_HotelCheckInNotifRQ)
  • provide the ResID of the test reservation (arrival date needs to be current PMS date)
  • check if the room status is on correct status
  • integ. partner sends a Check In
  • check the status of the test reservation in the PMS if it is Checked-In correctly
  • Cancel the test reservation
  • Test an incoming SystemDate look up
    (IO_SystemStatusRQ)
  • Partner sends the IO_SystemStatusRQ
  • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  • Test an incoming CheckOut
    (HTNG_HotelCheckOutNotifRQ)
  • Partner sends the HTNG_HotelCheckOutNotifRQ
  • check if the HTNG_HotelCheckOutNotifRS has been sent back without any errors and check if the reservation has been checked out correct in the PMS.
  • Test an incoming Folio Look Up
    (HTNG_HotelFolioRQ)
  • Partner sends the HTNG_HotelFolioRQ
  • check if the HTNG_HotelFolioRS has been sent back without any errors
  • Test an incoming Profile modification
    (OTA_ProfileModifyRQ)
  • Partner sends the OTA_ProfileModifyRQ
  • check if the OTA_ProfileModifyRS has been sent back without any errors and check if the modifications have been processed correct in the PMS
  • Test an incoming Profile creation
    (OTA_ProfileCreateRQ)
  • Partner sends the OTA_ProfileCreateRQ
  • check if the OTA_ProfileCreateRS has been sent back without any errors and check if the profile was created accordingly in the PMS
  • Test an incoming profile attach to a reservation

(IO_ResModifyNotifRQ)

  • Partner sends the IO_ResModifyNotifRQ
  • check if the IO_ResModifyNotifRS has been sent back without any errors and check if the profile was attached to the reservation accordingly
