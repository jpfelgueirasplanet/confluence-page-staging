Set Default Cash Register for incoming PostPayments

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up.

In order to register the incoming PostPayments with a specific cash register number inpAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

1. Define a default cash register in pAir

System data /POS/ Payment methods /POS

2. Add values that are used to define the default cash register for all IO-related postings in the configuration editor

Cash register definition

  • AddCashRegisteranddefaultfolder in Configuration/currentProperty/propData (Notice: "default" - low case letters !!!)
  • AddprotelIOvalue with Content type:intand set the default cash register number inDatatext box


Configure a KSK template for invoices

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelCheckOutNotifRS/TemplateNameInvoice

  • Create HTNG_HotelCheckOutNotifRS/TemplateNameInvoice folder if they do not exist
  • The Content-type for HTNG_HotelCheckOutNotifRS is null
  • The content-type for TemplateNameInvoice is String
  • The Data is the Name of the Template. Important: case sensitive!

  • The Invoice Template needs to be configured in the System Language

Note: 
Please always  use the templates out of "Meine Textvorlagen" only.

One way Installation


1

FYI: Before creating test reservations clarify with the property that you will do test reservations with additional ChargePosting in order that this could be canceled afterward so that the property is informed.

Prepare a test reservation ( departure date should be the installation date and preferably with a room amount of max €1) in advance that would be used to test the check-out functionality during the installation date.


Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs.

Examples

Steps for CheckIn workflow:

  1. IO_SystemStatusRQ to Protel
  2. OTA_ReadRQ to Protel (via ResID, CRS number or Name) 
  3. HTNG_HotelFolioRQ to Protel
  4. HTNG_ChargePostingRQ to Protel (Payments)
  5. HTNG_HotelCheckInNotifRQ to Protel

Optional: 
IO_ResModifyRQ ( Secom ca assign a beakfast manually).

OTA_HotelAvailRQ ( Checking the room availability)


EOD

Steps for CheckOut workflow:

  1. IO_SystemStatusRQ
  2. OTA_ReadRQ to Protel (via ResID,CRS number or Name)
  3. HTNG_HotelFolioRQ to Protel
  4. HTNG_ChargePostingRQ to Protel (invoice balance =0)
  5. HTNG_HotelCheckOutNotifRQ to Protel
