Installation Steps

 

General steps

Examples

1

Request authentication details from the Integration partner for the new installation and request them to send you the authentication details.

Please provide them the HotelID as well - internally for Protel this is the CloudID 

Data Security

Please have in mind that both vendor and IO interface token information is sensitive information that might be transported by Slack,
team viewer session in case of a 1:1 session, or with two-way authentication methods by other media. 

  • Authentication method using  a specific endpoint,  username and password given by the third party system for each hotel customer

General Preparation On-Premise

IO_Pre-Setup-Checklist_Partner_en.pdf

Add licence of I/O Module to system data

  • room status accepted for Check In

io_checkin_accepted_room_status

Depends on the customer

If there is no xsetup present for

pws | io_checkin_accepted_room_status | - 

then the C/I will be possible in the first clean/checked room.
If no room is available with this status, then C/I will fail with error RS to vendor


Note:
Checked AND cleaned is the default - the setting is only needed if ONLY checked rooms should be available for CheckIn.

pws | io_checkin_accepted_room_status | 1 
→ C/I is enabled only for clean rooms

pws | io_checkin_accepted_room_status | 4  
→ C/I is enabled only for checked rooms

One way Installation


1

PWS Configuration

  • enable LiveMode and activate ToggleTrigger for required message types for 1-way connection

Message type

LiveMode

ToggleTrigger

OTA_HotelResNotifRQ

2

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


3

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    for  the next one     year

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

Check for failed messages after the bulk upload: Manual Correction (Support-Tool Use Cases) 

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one.

2

  • Test an incoming Availability check
    (OTA_HotelAvailRQ)
  • please check if the OTA_HotelAvailRQ was responded successfully with an OTA_HotelAvailRS
  • Test an incoming Reservation Look Up (via ResID; via Arrival/DepartureDate and ResStatus)
    (OTA_ReadRQ)
  • Create a test reservation in PMS
  • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  • Test an incoming Check In
    (HTNG_HotelCheckInNotifRQ)
  • provide the ResID of the test reservation (arrival date needs to be current PMS date)
  • check if the room status is on correct status
  • integ. partner sends a Check In
  • check the status of the test reservation in the PMS if it is Checked-In correctly
  • Cancel the test reservation
  • Test an incoming SystemDate look up
    (IO_SystemStatusRQ)
  • Partner sends the IO_SystemStatusRQ
  • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  • Test an incoming CheckOut
    (HTNG_HotelCheckOutNotifRQ)
  • Partner sends the HTNG_HotelCheckOutNotifRQ
  • check if the HTNG_HotelCheckOutNotifRS has been sent back without any errors and check if the reservation has been checked out correct in the PMS.
  • Test an incoming Profile modification
    (OTA_ProfileModifyRQ)
  • Partner sends the OTA_ProfileModifyRQ
  • check if the OTA_ProfileModifyRS has been sent back without any errors and check if the modifications have been processed correct in the PMS
