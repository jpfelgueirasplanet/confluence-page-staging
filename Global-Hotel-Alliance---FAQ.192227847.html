Missing Mapping for the StayInfo

Background Information

With every EOD GHA is receiving the OTA_HotelStayInfoNotifRQ for all guests that are checked out on the current Business Day, and for all guests that are No-Show on the current Business Day. This OTA_HotelStayInfoNotifRQ is automatically created and pushed by the PWS. The ESB is checking in a second step, if the guests has a GHA Membership attached, if not, then the ESB is dropping the OTA_HotelStayInfoNotifRQ Message, because GHA is only interested in Reservation information for GHA Members.

Further Information for the OTA_HotelStayInfoNotifRQ that is created by the PWS is documented on the following Confluence - OTA_HotelStayInfoNotifRQ/RS


This OTA_HotelStayInfoNotifRQ is mapped to the STAY Message in the oxi-proxy. This mapping is documented on the following Confluence page, like all other GHA Mappings. - GHA Mapping

Special Configuration for GHA 2.0.

With the extension for GHA 2.0. GHA requests not only to map MarketCode, but also Transaction Accounts (TAA) and Methode of Payment. Therefore, we decided to use the Distribution Channel Settings (DCS) to make it as flexible as possible for hotels and partners.

The Distribution Channel Settings (DCS) can be enter in the FO → Reservation → Distribution Channel Settings


The partners/hotels can independently choose a number in the DCS and store the mapping for the GHA integration. Important this number needs to be added in the Installation (com.oracle.oxi) in BOB, because without this number the ESB does not know which mapping needs to be used.


To push the information from the DCS to the ESB, the PWS creates a IO_MappingNotifRQ out of the DCS information.

Important, to enable the message in general for a property,  the following XSetups need to be configured, so that the IO_MappingNotifRQ is sending the "DestinationServiceName"


Global - pws - iomappingdestinations - Name of the BOB Installation

With a ";" the service can be listed


As soon as this is configured, in the PWS the services can be selected and can be assigned to an IDS instance ID



If this is enabled, the PWS will push automatically with every change in the DCS an IO_MappingNotifRQ to the ESB. The ESB will store the information for the specific Hotel and Instances ID, and will automatically change the Values for MarketCode, TAA and Methode of Payment in the Stay Message for GHA.


How to check the Stay Message in Kibana?

In Kibana we always have the possibility to check "both" Stay Message, the OTA_HotelStayInfoNotifRQ, and the STAY message which is pushed to GHA.

In the "transaction-log" protocol, we will find the PWS message



In the "oxi" protocol, we will find the GHA message


What can be checked?

For a missing MarketCode/TAA/Methode of payment, the following things could be checked

  • is the Mapping in Protel (DCS) existing?
  • does the IO_MappingNotifRQ message contain a valid payload?
  • is the Mapping in the ESB existing? (please ping Constantin Stropsa to check this)
  • when was the last IO_MappingNotifRQ pushed to the ESB?


How to check the redemption workflow in Kibana

In order to be able to use GHA DISCOVERY 2.0, a hotel needs to have the respective connection to GHA which is "com.oracle.oxi" and also to GRAVTY, which is "com.gravtyoracle.ioxi". 

com.oracle.oxi

To ensure that all messages processed in protel (OTA, HTNG and IO) are successfully mapped against the oxi-protocol GHA is using, please make sure to always check the "transaction-log", as well as the "oxi-log" in Kibana when checking the message flow.

com.gravtyoracle.ioxi

For connectiong to GRAVTY, only OTA, HTNG or IO messages are used on both sides, and this is why only the "transaction-log" needs to be checked.

We also implemented three new message types for using with GHA 2.0:

  • IO_LoyaltyAccountInfoRQ/RS
  • IO_LoyaltyAccountInquiryRQ/RS
  • IO_LoyaltyAccountChargeRQ/RS

Please be aware that in order to proceed with the redemption, it should be a CHECKOUT procedure (starting with click on Check out)
As soon as the user opens the invoice in protel FO, the message "IO_LoyaltyAccountInfoRQ" is sent to GRAVTY.

This message asks for the current amount of DISCOVERY Dollars in a loyalty card. The message is answered with

  • AccountBalance → current amount of points
  • AccountChargeLowerLimit → minimum redemption value

As soon as the payment is confirmed (by clicking OK in the payment dialog), the points (DISCOVERY Dollars) will be redeemed and the message "IO_LoyaltyAccountChargeRQ" will be sent to GRAVTY.

Depending on the TransactionReference="XXXYYY", GRAVTY will void points. The TransactionReference="XXXYYY" is transmitted everytime a redemption takes place.
