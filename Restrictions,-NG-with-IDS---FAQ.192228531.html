Features from pAir NG which are supported in ARI updates through IDS

Advanced Rate Code

  1. SD/Rate Configuration/Rate/Rate code detail 

    1. Valid period => will affect OTA_HotelRatePlanNotifRQ and OTA_HotelAvailNotifRQ messages. No separate trigger for adding/modifying/removing a season. If valid period is set globally for the rate code - then the RoomRate season is ignored. Else if there is no season globally on the rate code - then each RoomRate is taken with its validity period. If there is no validity/season period difed neither on he global rate level nor on RoomRate level, then the validity of the rate is determined by the rest of rate configurations (e.g. rate strategies) and by the export period defined in the IDS as applicable for export. Multiple valid/season periods are as well supported  

      <StatusApplicationControl Start="2021-06-24" End="2021-06-26" InvTypeCode="KING" RatePlanCode="PTCS"/>
       Seasons should NOT overlap! Otherwise the rate calculation exported by the PMS will have errors in the calculation.


      <StatusApplicationControl Start="2021-06-24" End="2021-06-26" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <StatusApplicationControl Start="2021-06-28" End="2021-06-30" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <StatusApplicationControl Start="2021-07-02" End="2021-07-05" InvTypeCode="KING" RatePlanCode="PTCS"/>

    2. Valid for length of stay - FROM => sent out in OTA_HotelAvailNotifRQ 

      <LengthOfStay MinMaxMessageType="SetForwardMinStay" TimeUnit="Day" Time="3"/>

    3. Valid for length of stay - TO => sent out in OTA_HotelAvailNotifRQ 

      <LengthOfStay MinMaxMessageType="SetForwardMaxStay" TimeUnit="Day" Time="8"/>
    4. Valid for nights - FROM => N/A 
    5. Valid for night - TO => N/A 

    6. Valid for weekdays - ARRIVAL => OTA_HotelAvailNotifRQ 
      The respective date will be sent as closed for Arrival restriction
      There is no trigger to this restriction

      <AvailStatusMessage>
      <StatusApplicationControl Start="2021-06-29" End="2021-06-29" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <RestrictionStatus Restriction="Arrival" Status="Close"/>
      </AvailStatusMessage>

    7. Valid for weekdays - STAY=> OTA_HotelAvailNotifRQ 
      The respective date will be sent as Master Close restriction
      There is no trigger to this restriction

      <AvailStatusMessage>
      <StatusApplicationControl Start="2021-06-29" End="2021-06-29" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <RestrictionStatus Restriction="Master" Status="Close"/>
      </AvailStatusMessage>

      Valid for weekdays - STAY => OTA_HotelRatePlanNotifRQ 
      Will affect the @Start-@End date periods sent for the rate calculations

      <Rate Mon="true" Tue="true" Weds="true" Thur="true" Fri="true" Sat="true" Sun="true" Start="2021-07-11" End="2021-07-14" CurrencyCode="EUR" InvTypeCode="KING" MinGuestApplicable="0" MaxGuestApplicable="0">

  2. SD/Rate Configuration/Rate/RoomRates

    1. Valid period per room rate =>will affect OTA_HotelRatePlanNotifRQ and OTA_HotelAvailNotifRQ messages. No separate trigger for adding/modifying/removing a season. If valid period is set globally for the rate code - then the RoomRate season is ignored. Else if there is no season globally on the rate code - then each RoomRate is taken with its validity period. If there is no validity/season period difed neither on he global rate level nor on RoomRate level, then the validity of the rate is determined by the rest of rate configurations (e.g. rate strategies) and by the export period defined in the IDS as applicable for export. Multiple valid periods are as well supported.  
       Seasons should NOT overlap! Otherwise the rate calculation exported by the PMS will have errors in the calculation. 

    2. Valid for length of stay - FROM => sent out in OTA_HotelAvailNotifRQ 
      Can be configured to override the global rate restrictions if any are set. Or can be set per room rate.

      <LengthOfStay MinMaxMessageType="SetForwardMinStay" TimeUnit="Day" Time="5"/>

    3. Valid for length of stay - TO => sent out in OTA_HotelAvailNotifRQ 
      Can be configured to override the global rate restrictions if any are set. Or can be set per room rate.

      <LengthOfStay MinMaxMessageType="SetForwardMaxStay" TimeUnit="Day" Time="13"/>
    4. Valid for nights - FROM => N/A 
    5. Valid for night - TO => N/A 

    6. Valid for weekdays - ARRIVAL => OTA_HotelAvailNotifRQ 
      The respective date will be sent as closed for Arrival restriction
      There is no trigger to this restriction

      <AvailStatusMessage>
      <StatusApplicationControl Start="2021-06-29" End="2021-06-29" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <RestrictionStatus Restriction="Arrival" Status="Close"/>
      </AvailStatusMessage>

    7. Valid for weekdays - STAY => OTA_HotelAvailNotifRQ 
      The respective date will be sent as Master Close restriction
      There is no trigger to this restriction

      <AvailStatusMessage>
      <StatusApplicationControl Start="2021-06-29" End="2021-06-29" InvTypeCode="KING" RatePlanCode="PTCS"/>
      <RestrictionStatus Restriction="Master" Status="Close"/>
      </AvailStatusMessage>

      Valid for weekdays - STAY => OTA_HotelRatePlanNotifRQ 
      Will affect the @Start-@End date periods sent for the rate calculations

      <Rate Mon="true" Tue="true" Weds="true" Thur="true" Fri="true" Sat="true" Sun="true" Start="2021-07-11" End="2021-07-14" CurrencyCode="EUR" InvTypeCode="KING" MinGuestApplicable="0" MaxGuestApplicable="0">

Rate Strategies


The following rate strategies are exported with OTA_HotelAvailNotifRQ. New/modify/delete include a trigger for the API.

  1. "Close"  

  2. "Close to arrival" 

  3. "Minimum stay of X days" - by default is MinLOS for stay period 

  4. "Minimum stay of X days" on arrival 

    1. The Availability period set is applied on the @Start-@End time span - not on the week flags in the OTA messages

    2. Click on "Only on arrival" to have the strategy based on the date of arrival
