WrittenConfInst

For integration partners to be able to send confirmation/Modification/cancellation letters to their clients, they require the information held within the <WrittenConfInst> section of the OTA_HotelResNotifRQ

The following letter types can be sent via OTA_HotelResNotifRQ based on the templates with various letters:

  • Offers
  • Confirmations
  • Cancellations

A protel user can select the letter type that wishes to be sent, and this will populate the "WrittenConfInst" section of the OTA_HotelResNotifRQ message with one of the following corresponding code:

  • OFR
  • CNF
  • CXL

Add/change/remove profiles in a reservation

Adding/changing/removing profiles in a reservation is handled per their @Role 

(1) Reservation exists in the PMS with 1 guest profile A. This means the guest is both the booker and the arriving guest. Inbound Modification of the reservation is sent where only one guest profile is sent but it is a new guest profile B. As a result, the PMS will keep the old guest profile A as a Booker of the reservation and add the new guest profile B as an Arriving Guest

(2) Reservation exists in the PMS with 2 guest profiles: A for Booker and B for Arriving Guest. Inbound modification of the reservation is sent where only 1 guest profile is sent and its role is "ArrvingGuest" C. As a result, the existing profile A of the reservation is not changed. Old arriving guest B is replaced with the new arriving guest profile C

Inbound modification of the Reservation Comment using OTA_HotelResNotifRQ

With pAir release 21.28 it is now possible to add/modify the reservation comment on existing reservations using OTA_HotelResNotifRQ inbound.

Since the modifications of a reservation can be done by different partners, each of them can have a separate column. If the attributes are personalized per their originator - then each of the partners can manage their own comment section only. In order to avoid the PMS deleting the partner comment from the reservation or vice versa, the PMS front end will make the partner comment as read-only on the PMS UI. This requires the external comments to be labelled with a trailing sign: "##############################"

Sample inbound Comment section with vendor details provided:

Comment with vendor identifiers
<Comments>
<Comment CommentOriginatorCode="iHotelier" GuestViewable="true" Name="TC2">
<Text TextFormat="PlainText">test case 2 - add new comment 1 line</Text>
</Comment>
</Comments>

As a result, the follow up OTA_HotelResNotifRQ APIs will look like this:

Comment from PMS where inbound modification exist
<Comments>
<Comment CommentOriginatorCode="PMS-Frontend" GuestViewable="true">
<Text TextFormat="PlainText">Comment added from the PMS UI internally</Text>
</Comment>
<Comment CommentOriginatorCode="iHotelier" GuestViewable="true" Name="TC2">
<Text TextFormat="PlainText">test case 2 - add new comment 1 line</Text>
</Comment>
</Comments>

Inbound Rate details to reference multiple days of stay

A. Where partner provides the rate values:
There are 3 general ways how to reference multiple days of stay with the inbound OTA_HotelResNotifRQ msg depending on the purpose. In general - each <RoomRate> element shows the Rate Plan Code detail (thus if it is the same for the whole stay - only 1 such is enough) and the <Rate> element shows the Rate Amount values (thus if they are the same for the whole stay - only 1 such is enough)

  • UC1: When the Rate Plan Code is identical and the Rate Amount values are identical for all days of stay - then it is only needed to have 1 RoomRate and 1 Rate elements
  • UC2: When the Rate Plan Code is identical but the Rate Amount values are different between the days of stay - then it is required to have 1 RoomRate and multiple Rate elements 
  • UC3: When the Rate Plan Code is different between the days of stay - then it is required to have multiple RoomRate elements - at least 1 for each distinct Rate Plan Code

B. Where partner does not know the rate values:
If the partner wants to rely on the protel PMS rate details setup instead of fetching them from a push message, then they can send an empty RoomRate element. In this case the PWS will query itself for the rate setup i nthe PMS and will pick up the current setup for the provided room type-rate plan-date combination.

Note: If this is a modification to a reservation and not a commit - anything the hotel did as manual edits to the rate values on the reservation will be lost with such reservation import.

C. Where partner does not want to change the rate values:
If the partner does not want to change the rate details - but smth diff on the reservation and has not enough information to pass the rate details correctly to the PMS, they can send a flag in Rate/@CachedIndicator="true". If at least one such exists - no Rate details will be read by the PMS service during the reservation import. Instead - the existing settings of the reservation rate details in the PMS will not be touched. 

Note: Applicable for transaction action "Modify" only.
