Circular Message Flow

The circular message flow prevents a message loop between Integration Partner and Protel.

It is applicable for message types which the PMS accepts in both directions In/Out (e.g. OTA_ProfileModifyRQ, OTA_HotelResNotifRQ, etc.)


  • Message Flow
  • How to check a Circular Message Flow in Kibana
  • How to check a Circular Message Flow for the PWS

Message Flow

The Integration Partner sends an OTA_ProfileModifyRQ message to Protel. This profile is modified in the PMS and then sent again as OTA_ProfileModifyRQ by the PMS. The Integration Partner will also modify the profile again and send it back to Protel, thus creating an endless loop. 





To stop this endless loop, we have implemented the "Caused-By CorrelationID". The "Caused-By CorrelationID" is automatically added to the outbound messages by the PMS if a modification has been made due to an inbound message. Based on the "Caused-By CorrelationID", our ESB knows to which Integration Partner the message does not have to be sent. 



How to check a Circular Message Flow in Kibana

As soon as the ESB is receiving a Message with the "Caused-By CorrelationID" the ESB will automatically not sent the message to the "originator" and will put in Kibana a message "Dropped due to circular message flow detection" for this specific Integration Partner.

<?xml version='1.0' encoding='utf-8'?>
<env:Envelope xmlns:env="http://www.w3.org/2003/05/soap-envelope">
<env:Header>
<htnga:CorrelationID xmlns:htnga="http://htng.org/PWSWG/2007/02/AsyncHeaders">61181c7c-819b-4a4f-b324-5cf5ea01d874</htnga:CorrelationID>
<Source xmlns="http://protel.io/soap" Product="protel On Premise SPE" ProductVersion="21.05000" Module="protel.WebServices" ModuleVersion="3.4.10.10955" Service="io.protel.onpremise"/>
<CausedBy xmlns="http://protel.io/soap">17e92a07-b52a-47a1-8a30-8847524eeae1-1639729716025</CausedBy>
</env:Header>
<env:Body>


How to check a Circular Message Flow for the PWS


1. Inbound Message from Integration Partner A, profile is updated in the PMS, new entry in the pws_jobs is created

→ causedBycorreletationid is filled with the inbound CorrelationID

→ client = 2

2.  kunden trigger fires and increments triggercount to 1

→ clientId changes to 0 (NULL or 0 means preparer is ready to grab this entry and transfer it to pws_livejob)

3. Preparer in the pws_livejob grabs the pws_jobs entry

a) If the triggercount is <= 1 there was no further change from front office on the kunden row, so the causedBy Mechanism is used

b) If the triggercount is > 1 there were additional changes on the kunden row (from frontoffice) which is maybe important for Vendor A. CausedBy mechanism is not used.

4a) Preparer writes the pws liveJob entry into pws_livejob with causedBy CID inside the JSON column

4b) Preparer writes the pws liveJob entry into pws_livejob without causedBy CID inside the JSON column

5) Message is processed with causedBy element and ESB avoid sending it to vendor A
