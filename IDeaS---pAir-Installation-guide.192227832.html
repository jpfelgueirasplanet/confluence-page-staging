Installation Steps

 

General steps

Examples

1

Check the active lists for reservations AND allotments for entries without market code for the following 2 years
and let the customer to add the missing ones BEFORE starting the Bulk Upload!!

(IDeaS only accepts reservations and allotments with market code - if you don't correct it, there will be discrepencies in the DQE)

Reservations:
--> add the MarketCode column to the reservation list
--> order by MarketCode
--> there are empty entries? Let the customer fill the missing ones.

Allotments:
--> you (or the customer) need to go through the allotments one by one - there is no easier way unfortunately.


2

You don't need to request authentication details for the new installation (they are always the same).

You need to request the ChainCode in advance.

Please provide them the HotelID (HotelCode) - internally for Protel this is the CloudID.

  • Authentication method using the same Endpoint as default and always the same User and Password

    EP: https://ngi.ideas.com/ngipublic/ws/htng
    User: protelio
    PW: Y(qhL5vcEF!pGByKR*x#UA=

3

General Preparation for pAir General (pAir)


4

Enter the ChainCode in Config Editor

currentProperty/propData/Interfaces/IO/Gateway/HotelChainCode
currentProperty/propData/Interfaces/IO/GatewayBTS/HotelChainCode


5

Activate the MarketCode to be mandatory (required) for reservations (XsetUp)

6

Bob installation: one for pAir (if not existing) and one for the Integration  + one for SCS

  • provide the token corresponding to the Integration to the integration partner 


 

One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


2

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ) for the last years and ending with 31st Dec 2099
  • for blocks (OTA_HotelInvBlockNotifRQ) for the last 2 years and ending with 31st Dec 2099

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


3

Send Statistics

--> Trigger the OTA_HotelStatsNotifRQ once per type (needed by IDeaS to do the DQE on their side)

--> after Bulk always 730 days past / 365 days future
--> each of the mentioned URL's - one by one

Log in to the pCC (System Data) for the specific HotelID and open an additional browser tab to post the URLs (not yet working with Next Gen - pCC Log in needed!):

Future
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/true?daysForth=365&step=3
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/false?daysForth=365&step=3
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/false?daysForth=365&segment=true&step=3

Past
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/true?daysBack=730&step=3
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/false?daysBack=730&step=3
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/false?daysBack=730&segment=true&step=3

pattern:
http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/stats/{true/false}?{daysBack/daysForth}=XXX&step=3

See pAir | Triggering messages: API Calls for further explanation


4

If requested, you can change the number of days for past and future OTA_HotelStatsNotifRQ messages triggered by EOD.
To do so, you need to enter the number of days in BOB under Environment variables for the pAir installation of the hotel:

  • IO.Gateway.OTA_HotelStatsNotifRQ.StatsDaysForth

  • IO.Gateway.OTA_HotelStatsNotifRQ.StatsDaysBack

5

Please inform IDeaS that your done with the Bulk Uploads and sending the Statistics!


6

Data Validation Report - DQE

--> is done on IDeaS side, no report needed from our side

  • The Integration partner will process the DQE and send you a report afterwards in case of discrepancies


 

Two way Installation

Examples

1

General information about the 2-way activation

IDeaS Installation team will inform the Integrations Support Team / Distribution Partner about the scheduled date for the initial upload of decisions.

Integrations Support Team / Distribution Partner need to respond back with a certain time for a call together with IDeaS which is then meant for doing the test of the upload together. IDeaS will send an invitation for the call.


The customer will be informed only as soon as the uploads have been successfully tested together with IDeaS (upload itself and some Cross Checks of dates). The property will be advised by IDeaS to do checks on their own to guarantee the correctness of the data.

Before handing the data over to the customer a fresh set of StatsNotif messages need to be sent from protel side to trigger a new decision upload on IDeaS side.
(if this is necessary depends on how long ago the last Statistics have been sent).


2

Test an incoming Overbooking decision

(OTA_HotelAvailNotifRQ)

--> stored in FO - Options - Overbooking by PWS per room type and on hotel level

--> will be received inside an inbound OTA_HotelAvailNotifRQ message
MessageContentCode="1" --> Overbooking room type level
MessageContentCode="7" --> Overbooking hotel level

--> negative values in the messages will be corrected to positive ones

3

Rates are updated by the Integration partner

(OTA_HotelRatePlanNotifRQ)

--> will be received inside an inbound OTA_HotelRatePlanNotifRQ message

--> in this case IDeaS will send rate values per room type & occupancy which are displayed in DailyRates module (Reservation -> Daily Rates)
--> you need to activate the Daily Rates per XSetUp

--> IDeaS and the customer will agree on a rate code which should be filled with values from IDeaS side
--> please make sure that the checkbox "Yieldable" of this rate code is activated

4

Test incoming Rate decision

(OTA_HotelRatePlanNotifRQ)

--> AdditionalGuestAmounts from the message are also supported - those will be added to the occupancy up to 6 adults
--> Child Rates are not supported as DailyRates could not store rate values for children

-->the customer must not influence the calculation of the desired yieldable rate in the DailyRates by adding prices or formulas manually. The following error occurs otherwise: "ShortText="mixing modes not possible: 2!=OverrideWithAbsoluteValue DailyRate(xxx)"

How to create SCS jobs for IDP sending

Details

IDeaS is scheduling certain IDP times with the customers (in most of the cases).

At these times pAir needs to send additional statistics exports (delivered with OTA_HotelStatsNotifRQ).

pAir needs to get triggered to start building and sending those messages. 
This triggering is done by the SCS (Scheduled Command Service).

That's why you need to create jobs in the SCS if IDeaS informs you about the IDP (Intra Day Processing) times.

Verify in Kibana that the outgoing message (OTA_HotelStatsNotifRQ)  is answered with RS message from the Integration system
