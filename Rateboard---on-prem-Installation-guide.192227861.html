One way Installation


1

PWS Configuration

  • Activate the Trigger and LiveMode for required message types for 1-way connection& additionally the Global Live Mode.

Message type

ToggleTrigger

LiveMode

OTA_HotelResNotifRQ

OTA_HotelInvBlockNotifRQ

OTA_HotelInvCountNotifRQ

2

Rateboard is not interested in profile data, so you can exclude profile data from the OTA_HotelResNotifRQ with this setting

Please note This should only be done if no other system is connected via I/O Integration which needs profile data.

3

Rateboard is expecting to receive an extended time span of OTA_HotelInvCountNotifRQ export. You can set this up in PWS dialogue directly. 
Please change the value to 770 days. 

Note: Please make sure you selected the correct hotel in PWS dialogue. The entry is then set automatically in System Data X-Setup.

4

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


5

Testing Occupancy

(OTA_HotelInvCountNotifRQ)

  • trigger for 1 year the OTA_HotelInvCountNotifRQ once on the server of the customer where the PWS service is running 
    • If today is 01st January 2020 - start date would 01/01/2020
    • End date: 31.12.2020
  • Verify in Kibana that the outgoing messages OTA_HotelInvCountNotifRQ is answered with RS message from the Integration system
  • for Occupancy the PWS will send the OTA_HotelInvCountNotifRQ automatically 3 times a day at 5,14 and 22 o'clock

Open a browser to post the URLs:
(NOTE - you have to change some details here!)

pattern:
http://IP-address(PWS):Port(PWS)/ota/hotelinvcountnotif/{MPE_hotel}/send/{START:YYYY-MM-DD}/{END:YYYY-MM-DD}

e.g.: http://127.0.0.1:8046/ota/hotelinvcountnotif/1/send/2020-01-13/2020-12-31


WITH PWS BUILD 3.4 AND HIGHER:

http://127.0.0.1:8000/ota/hotelinvcountnotif/{MPE_hotel}/send/{NUMBER OF DAYS}

 e.g.: http://127.0.0.1:8012/ota/hotelinvcountnotif/3/send/365

6

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)

    • Start date: 1st of running month 2 years ago (if today is 6th March 2019 - start date would be 01/03/2017)

    • End date: 31/12/2099

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

Check for failed messages after the bulk upload: Manual Correction (Support-Tool Use Cases) 

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


7

Data Validation Report - DQE

  • after checking if the bulk successfully finished, provide the integration partner the report
  • The report is to be found in the tab Add-Ons of the front office
  • after checking if the bulk successfully finished, provide the integration partner the report
  • Use the tool for the dqe https://app.rateboard.io/login to compare the data
  • Login in the link
  • Choose the hotel

  • Get an overview of the reservations in RB

  • There is a button "compare data"

  • Upload the DQE report from protel, follow the settings on the screen and you'll get a pdf or xls with the comparison. 


8

1-way installation is completed with this point

  • The customer should clarify with Rateboard which rate codes need to be updated by the integration partner when activating 2-way

  • Please send a mail to the customer and let them prepare the rate code which should be filled by Rateboard providing this link:
    Rate set up for inbound Daily Rates (on premise)

4

Test incoming Rate update

(OTA_HotelRatePlanNotifRQ)

  • the integration partner should ask the customer which rate code should be updated
  • make sure that the rate which needs to be updated is set to yieldable
  • ask the integration partner to push some rates
  • ask the customer to check the data via FO - Add Ons - Daily Rates - Enter daily rates editor
  • protel database table is ckit_dailyrates_rates
  • will be received inside an inbound OTA_HotelRatePlanNotifRQ message
  • AdditionalGuestAmounts are also supported - those will be added to the occupancy up to 6 adults
  • Child Rates are not supported as DailyRates could not store rate values for children

5

Prepare incoming Restriction Test

  • If the restrictions sent for the baserate by Rateboard should also update the restrictions of the depending virtual rate codes you need to activate the yellow marked check box. Otherwise NO restrictions for the depending virtual rate codes will be entered into protel!



6

Test incoming Restrictions 

(OTA_HotelAvailNotifRQ)

  • ask the Integration partner to clarify with the customer to have the Test rate code invalidated (through expired validity) after the verification of the Restriction message is done
  • Make sure that the rate which needs to be updated is set to yieldable. Ask in the channel for which day the rate has been pushed, check the date and send a screenshot from the Protel Daily Rates if the rate for the specific date is showing in the list. 
  • Ask the Integration Part in the Slack channel of the integration to push a restriction for a specific day on the Test rate code and then check in the Price availability for the given date if the restriction has been set. 

7

Update the protel SMP page with the information about the installed customer

Development SMP page / Development / Services / Prospect Hotels / Add
