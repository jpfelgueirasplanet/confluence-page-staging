One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


2

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    for future reservations (next 2 years)

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


3

Data Validation Report - DQE

  • The DQE is NOT mandatory, only in case of obvious discrepancies a check should be done together with the Duve Support team.


  • after checking if the bulk successfully finished, provide the integration partner the report
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Reporting of the front office in pAir.

 

Two way Installation

Examples

1

General Workflow

  • Reservation set up in protel - Reservation sent to Duve (OTA_HotelResNotifRQ)
  • Pre-Stay E-Mail sent to guest with URL for Duve App/Portal
  • Guest connects to Duve URL


BEFORE CHECK IN

  • Guest changes profile details - Duve -> protel (OTA_ProfileModifyRQ)
  • Guest checks invoice (HTNG_HotelFolioNotifRQ / HTNG_HotelFolioRQ) (Certified, but not yet implemented by Duve)
  • If activated, guest makes deposit payment - Duve -> protel (HTNG_ChargePostingRQ)
  • Guest can purchase add-ons (added as fixed charge) and if activated, make payment (payment to invoice) for these add-ons Duve -> protel (IO_ResModifyRQ / HTNG_ChargePostingRQ)
  • Guest can add arrival and departure time - Duve - protel (IO_ResModifyNotifRQ)
  • Guest can change the room type (upsell), if activated - Duve → protel (Room type change and Revenue Posting as fixed charge via IO_ResModifyRQ)
  • Duve sends information as trace to protel Duve - protel (IO_ResModifyNotifRQ)


CHECK-IN (on day of arrival)

  • Guest can check in via the app - Duve -> protel (HTNG_HotelCheckInNotifRQ)  (Certified, but not yet implemented by Duve)


IN-HOUSE

  • Guest can purchase add-ons and if activated, make payment for these add-ons Duve -> protel (IO_ResModifyRQ / HTNG_ChargePostingRQ)


CHECK-OUT (on day of departure)

  • Guest can check out via the app if the invoice is balanced -  Duve -> protel (HTNG_HotelCheckOutNotifRQ)  (Certified, but not yet implemented by Duve)


2

Test an incoming Reservation 

(OTA_HotelResNotifRQ)

  • ask the Integration partner to cancel the reservation after the verification is done.
  • Important: check if the integration needs reservations with Credit Card, and create a test reservation with CC details. Verify if the details have been correctly saved in the PMS.


3

CheckIn Workflow

1. Test an incoming Profile modification → (OTA_ProfileModifyRQ)
 Duve is not saving a profile if it is not attached to a reservation 

 2. Test incoming Trace → (IO_ResModifyNotifRQ)

3. Test incoming arrival time → (IO_ResModifyNotifRQ)

Reopen the app in the browser

4. Test purchase add-on →(IO_ResModifyNotifRQ)

4. Test an incoming invoice detail request → (HTNG_HotelFolioRQ)  NOT IMPLEMENTED YET BY DUVE

5. Test an incoming Payment posting → (HTNG_ChargePostingRQ) (if payment activated)

5. Test room upgrade, if activated → (IO_ResModifyNotifRQ)

6.  Test an incoming Check-In →  (HTNG_HotelCheckInNotifRQ) NOT IMPLEMENTED YET BY DUVE

Void the payment & Cancel the test reservation afterwards


4

CheckOut Workflow (Departure must be on current day)

  1. Test an incoming invoice detail request → (HTNG_HotelFolioRQ)  NOT IMPLEMENTED YET BY DUVE

2. Test an incoming Payment posting → (HTNG_ChargePostingRQ)

3. Test an incoming Check-Out (HTNG_HotelCheckOutNotifRQ) NOT IMPLEMENTED YET BY DUVE

Void the payment/revenue once finished the test.


5

Update the developer portal with the information about the installed customer
