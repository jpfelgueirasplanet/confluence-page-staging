Installation Steps

 

General steps

Examples

1

websLINE uses the same connection details for each new installation. EP; basic User/Pass are set as default values in the manifest

Please reach out to websLINE via slack or with an email to systems@websline.com to provide them the HotelID (protel Cloud ID) and token of the new installation 

2

The following System Data of the Customer PMS must be provided to websLINE before sending any API messages to them:

  • Room Type Codes (Settings/Hotel Management/Room Types)
  • Rate Plan Codes (Settings/Rate Configuration/Rates)
  • Transaction Account Codes (Settings/Financial Configuration/Items)
  • Deposit payment method (Settings/Financial Configuration/Payment Methods)
  • Market Codes (Flash SD/Reservations/Market Codes)
  • Source of Business Codes (Flash SD/Reservations/Source codes) 

Take a print screen of those reports and provide to the partner. Once done - delete the locally stored copies of data.

 It would take the partner approximately 30 minutes to import/map the PMS SD codes and have the codes usable with their booking engine.

 

3

 Confirm with the Customer about the interface:

  • Which Room Types should be mapped for the booking engine (to be used on inbound reservations and rate updates)?
  • Which Rate Codes should be mapped for the booking engine (to be used for inbound reservations and rate updates)?
  • How should the deposit payment be posted - as revenue or as payment and to which TAA code/Payment Method?
  • websLINE posts deposit payments. All payments done will get posted to the cashier no. - which cash register ID should be used for the payment postings?
    Please make sure that the cashier no. really exists
  • Which fixed charges will be applied on the booking engine (to be used with inbound reservations - fixed charges)?
  • What children age ranges should be configured in IDS (e.g. "Child 1" corresponds to what child age range)?

 Confirm with the Partner about the needed IDS matchings (RoomType / RateCode)

 If the Customer connection was initiated from websLINE's end, the list has already been aligned between them - thus protel support can obtain the codes to map from the partner. Else - if it is a protel Customer - obtain the needed codes from the Customer in advance to enable the installation testing.



4

General Preparation for pAir General (pAir)



5

Bob installation: one for pAir (if not existing), one for the Integration and one for IDS

  • provide the token corresponding to the Integration, to the integration partner (ex. screenshot token starting with K225) 
  • IDS installation

Leave the default values for URL (EP) and authentication (User/Pass) the way they are with default values. From the drop-down menu for ResTarget - choose "protel Air"

6

Special configurations:

Set a default cash register no. for websLINE deposit postings

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up. 

In order to register the incoming Deposit Payments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming payments through the HTNG_ChargePostingRQ.

  1. Define a default cash register in pAir

System data / POS / Payment methods / POS

 The respective menu in NG is Settings/Financial Configuration/Cash Registers

2. Add an xsetup that is used to define the default cash register for all IO-related postings in the configuration editor

  • Add CashRegister and default folder in Configuration/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box

7

Configuring an IDS Instance for I/O: How to configure a protel I/O IDS

  • Reaching the pAir server on Prod
  • Add a new instance
  • Configuring instance  No reservation import email notifications are needed
    • Export  Set export span for 1,5 years (18 months) in future, every 5 minutes
      • Rate Values - OTA_HotelRatePlanNotifRQ
        • Enable the rate export in tab Export/Rate Vlues
        • Configure the Children in tab Common/Children age ranges - To be confirmed with partner/Customer
      • Restriction Values - OTA_HotelAvailNotifRQ
        •  Only enable the restriction export in tab Export/Restriction Values - no further configuration required
      • Deactivate the export for OTA_HotelInvCountNotifRQ from IDS (the message is sent from pAir)
  • Create mapping 
    • mpehotel
    • RoomType
    • RateCode

 Note: If automated matchings are used in the IDS configuration, then the IDS will create the matching for the respective category itself. But this will disable the option to do any adjustments on the auto-matched list. Thus, it is recommended to do the matchings manually

     

 

One way Installation


1

a) Using manual trigger link, send IO_StatsNotifRQ for 45 days in past and 365 days in future

b) Verify in Kibana that the outgoing message is answered with IO_StatsNotifRS message from the Integration system

pAir | Triggering messages: API calls:

http://connect.protel.net/baoBackline/resources/interfaces/io/trigger/stats?step=3&daysForth=365

http://connect.protel.net/baoBackline/resources/interfaces/io/trigger/stats?step=3&daysBack=45

2

a) Using manual trigger link, send OTA_HotelInvCountNotifRQ for 365 days in future

b) Verify in Kibana that the outgoing message is answered with OTA_HotelInvCountNotifRS message from the Integration system

pAir | Triggering messages: API calls:

http://connect.protel.net/baoBackline/resources/interfaces/htng/trigger/InvCount/365?step=3&sum=true

3

a) Using IDS/Start full data export for 1,5 years (18 months) in future for Rates and Restrictions

 Could be adjustable per Customer

b) Verify in Kibana that the outgoing messages are answered with OTA_HotelRatePlanNotifRS and OTA_HotelAvailNotifRS respectively from the Integration system

Right-click on the IDS instance line / Start full data export:

4

Create a test reservation in the future (at least one year after the system date)

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 
  • always provide the reservation number to your integration partner, so that the reservation can afterwards be cancelled


5

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ) for the last 1 year and everything in future

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


6

Data Validation Report - DQE

DQE is evaluated in terms of Gross/Brutto values

  • after checking if the bulk successfully finished, provide the integration partner the report
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Office (BÃ¼ro) of the front office in pAir.


 

Two way Installation

Examples

1

FYI: Before creating test reservations clarify with the property that you will do test reservations with additional ChargePosting in order that this could be canceled afterwards so that the property is informed.

  1. Test an incoming Reservation (OTA_HotelResNotifRQ)

    a) Ask the partner to create a test reservation in their booking engine for at least 1 year in future

    b) Verify in Kibana that the OTA_HotelResNotifRS from pAir is present with <Success> and no warning/error.

    c) Verify in Kibana that the OTA_HotelResNotifRQ from pAir is present and responded with OTA_HotelResNotifRS with <Success> and no warning/error

    d) Verify in the PMS UI that the reservation details are captured properly

    e) Ask the partner to cancel the reservation from step (a) and repeat the same (b-d) steps afterwards

  2. Test an incoming Deposit Payment (HTNG_ChargePostingRQ)

    a) Ask the partner to create a test reservation in their booking engine for at least 1 year in future with included deposit payment

    b) Check HTNG_ChargePostingRQ/RS workflow in Kibana log

    c) Check the reservation folio for the proper deposited amount posting

    d) In order to void the posting - the partner needs to send the reversed deposit amount

  3. Test an incoming Fixed Charge (IO_ResModifyNotifRQ)

    a) Ask the partner to modify the reservation by adding a Fixed Charge

    b) Check the fixed charge posting in the PMS UI

    c) Check the IO_ResModifyNotifRQ/RS workflow in Kibana log


  4. Test an incoming Guest Profile modification (OTA_ProfileModifyRQ)

    a) Ask the partner to modify data on the guest profile from the reservation

    b) Check the change is reflected in the PMS UI

    c) Check the OTA_ProfileModifyRQ/RS workflow in Kibana log

 No CC data import is supported

  • Make sure that any test reservations are cancelled after the verifications are done.
