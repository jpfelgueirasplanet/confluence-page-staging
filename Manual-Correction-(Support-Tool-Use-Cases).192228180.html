Manual Correction (Support-Tool Use Cases)

  • Failed messages after the bulk upload
    • Failed messages with status=3
      • Missing Profiles
      • Reservations without ResUser and no change in table "aktion"
    • Failed messages with status=5
    • Wrong xml payload
    • e.g. OTA_CreateProfileRQ failed after Bulk Upload
  • Before the Bulk Upload is created


! IMPORTANT !

With PWS release >= 2.13.x the manual correction is not needed BEFORE Bulk Upload anymore (cases are handled by PWS automatically), only the last 2 cases could still happen and need to get checked (Wrong xml payload OR failed messages due to vendor side not being available during bulk upload).

Failed messages with status=5

tbc

Wrong xml payload

The error message is like "HTTP=NotAcceptable...Failed to detect operation name - please check the payload", i.e. points to reasons where a content of an xml attribute/s is breaking the validity of the payload. For example, this could be due to: special/unidentified characters, line breaks

Troubleshooting:

  1. Load the API for the reservation with the url trigger. (on prem | Triggering messages: API calls (PWS))
  2. Validate the payload with an xml editor to allocate the place where the integrity of the payload is breaking and the very attribute content causing it.
  3. Allocate the attribute in the PMS UI or DB table and edit the content manually to avoid the character/content that is in error.
  4. Proceed with re-running the failed reservations from the bulk with the below SQL queries.

e.g. OTA_CreateProfileRQ failed after Bulk Upload

  1. Check how many of these profiles failed
select * from pws_btsmessagejob where status=5 and knownmessage='OTA_ProfileCreateRQ' and pwsbtsjobid=xy
select * from pws_btsmessagejob where status=5 and knownmessage='OTA_ProfileModifyRQ' and pwsbtsjobid=xy


Vendor side was not available while sending messages


2.  Delete the idvalues out of the pws_hashs

delete from pws_hashs where knownmessageid in (select idvalue from pws_btsmessagejob where status=5 and knownmessage='OTA_ProfileCreateRQ' and pwsbtsjobid=xy)
delete from pws_hashs where knownmessageid in (select idvalue from pws_btsmessagejob where status=5 and knownmessage='OTA_ProfileModifyRQ' and pwsbtsjobid=xy)

3. Insert into pws_jobs from tables kunden

If the Profile message should be sent out as an "OTA_ProfileCreateRQ", please use 10 as event_type


INSERT INTO pws_jobs (clientid,state, isbulk, event_type, event_value, event_param, mpe, retries)
select 0 as clientid, 1 as state, 0 as isbulk, 10 as event_type, kdnr as event_value, mpehotel as event_param, mpehotel as mpe, 0 as retries
from kunden where kdnr in (select eventvalue from pws_livejob where knownmessage = 'OTA_ProfileCreateRQ' and updated >= '2019-08-21' and status >= 5)


If the Profile message should be sent out as an "OTA_ProfileModifyRQ", please use 12 as event_type

INSERT INTO pws_jobs (clientid,state, isbulk, event_type, event_value, event_param, mpe, retries)
select 0 as clientid, 1 as state, 0 as isbulk, 12 as event_type, kdnr as event_value, mpehotel as event_param, mpehotel as mpe, 0 as retries
from kunden where kdnr in (select eventvalue from pws_livejob where knownmessage = 'OTA_ProfileModifyRQ' and updated >= '2019-08-21' and status >= 5)

Before the Bulk Upload is created

  1. check for profiles with no profile number, update respectively (for the last 2 years):
    - select kundennr,buchnr, *from buch / buchold / stobuch where kundennr=-1 and datumvon >= '2017-01-01' and mpehotel=xy
    - update buch / buchold / stobuch set kundennr = Kundennr von Deleted Profil where kundennr = -1 and datumvon >= '2017-01-01' and mpehotel = x
  2. check for empty resuser (for the last 2 years):
    select resuser, *from buch / buchold / stobuch where resuser = '' and mpehotel = xy and datumvon >='2017-01-01'
    update buch / buchold / stobuch set resuser = 'unknown' where resuser = '' and datumvon >='2017-01-01' and mpehotel=xy
