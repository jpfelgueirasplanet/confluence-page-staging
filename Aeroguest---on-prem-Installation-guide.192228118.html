10

BAR Rates in the  OTA_HotelAvailRS Message

BAR rates returned in OTA_HotelAvailRS are in line with those in the F11 BAR mode and WBE. The message will display the lowest rate per room type from the BAR group, irrespective of whether it is a virtual or base rate. The message will not display all rates assigned to the BAR rate group. If a rate is required that is not present because it is not the lowest current rate in the BAR Group, then this rate must be moved from the BAR group into a standard group

12

Errors regarding the HTNG_ChargePostingRS

  • The third-party would not be able to process the payment because it took more than 30s for the HTNG_ChargePostingRS to be sent out. In Kibana The HTNG_ChargePostingRS will include a success

 To mitigate the problem the following configuration settings are suggested as may  reduce the comserver load

  1. Make sure PWS is connected to its own taskserver
  2. The new taskserver should have nothing else running against it, no ckits, no IDS
  3. IDS should be pointed to its own taskserver and not the comserver
  4. Taskserver should be set to reload system data cache every 60 minutes
    TaskServer_Setup
  5. Set this on taskserver.ini as that is where the pws is pointing
    [system]
    debugevtimerexecutiontimer=1
    thresholdevtimer=900
  6.  Make sure passerby account is configured and available
  7. Add the xsteup on global level
    ComServer | Whitelist | GetComputerName 
  8. If there is a timeout or error, check what the taskserver displays on the screen.

Kiosk Workflow overview 

Steps for CheckIn workflow:

  1. IO_SystemStatusRQ to Protel
  2. OTA_HotelAvailRQ to Protel
  3. OTA_ReadRQ to Protel (via ResID or DateType/ResStatus) DateType/ResStatus is used to retrieve a list of reservations with a certain ResStatus
  4. IO_ResModifyNotifRQ to assign a RoomID if no room is already allocated for the reservation. 
  5. HTNG_HotelFolioRQ to Protel
  6. HTNG_HotelCheckInNotifRQ to Protel
  7. HTNG_ChargePostingRQ to Protel


Steps for CheckOut workflow (after EOD has been processed) :

  1. IO_SystemStatusRQ
  2. OTA_ReadRQ to Protel 
  3. HTNG_HotelFolioRQ to Protel
  4. HTNG_ChargePostingRQ to Protel (invoice balance =0)
  5. HTNG_HotelCheckOutNotifRQ to Protel


Apart from the CheckIn and CheckOut flow, Aeroguest is also able to send OTA_ProfileModifyRQ's in order to modify e.g. phone no., nationality, name, email and/or address. 


3

  • Test an incoming SystemDate look up
    (IO_SystemStatusRQ)
  • Partner sends the IO_SystemStatusRQ
  • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  • Test an incoming Reservation Look Up via ResID first and after success via DateType/ResStatus (DateType/ResStatus is used to retrieve a list of reservations with a certain ResStatus)
    (OTA_ReadRQ)

via ResID:

  • Create a test reservation in PMS
  • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success

via DateType/ResStatus:

  • Create a test reservation in PMS
  • provide the DateRange and the ResStatus to the Int. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  • Test an incoming Availability check (Aeroguest needs this in order to have the available rooms for a room assignment in the IO_ResModifyNotifRQ step)
    (OTA_HotelAvailRQ)
  • please check if the OTA_HotelAvailRQ was responded successfully with an OTA_HotelAvailRS
  • Test an incoming reservation modification
    (IO_ResModifyNotifRQ)

Assigning a room to a reservation:

  • provide to the Int. partner a vacant clean/checked RoomID if not already happened via OTA_HotelAvailRS.
  • please check if the IO_ResModifyNotifRS has been sent back without any errors and check if the reservation got assigned to the correct room in PMS

Fixed charge using an article:

  • provide to the Int. partner the code of an article, if not already happened from the mapping values and let him post the article on the invoice of a reservation.
  • please check if the IO_ResModifyNotifRS has been sent back without any errors and check if the article has been posted correctly in the PMS   
  • Test an incoming posting
    (HTNG_ChargePostingRQ)
  • provide the ResID of the test reservation
  • integ. partner posts an amount on the folio (i.e. 10 Euro)
  • open the folio in the PMS and check if the amount was posted correctly
  • Test an incoming Check In
    (HTNG_HotelCheckInNotifRQ)
  • provide the ResID of the test reservation (arrival date needs to be current PMS date)
  • check if the room status is on correct status
  • integ. partner sends a Check In
  • check the status of the test reservation in the PMS if it is Checked-In correctly
  • Cancel the test reservation
  • Test an incoming Profile modification
    (OTA_ProfileModifyRQ)
  • Partner sends the OTA_ProfileModifyRQ
  • check if the OTA_ProfileModifyRS has been sent back without any errors and check if the modifications have been processed correct in the PMS 
  • Test an incoming Folio Look Up
    (HTNG_HotelFolioRQ)
  • Partner sends the HTNG_HotelFolioRQ
  • check if the HTNG_HotelFolioRS has been sent back without any errors
  • Test an incoming CheckOut
    (HTNG_HotelCheckOutNotifRQ)
  • Partner sends the HTNG_HotelCheckOutNotifRQ
  • check if the HTNG_HotelCheckOutNotifRS has been sent back without any errors and check if the reservation has been checked out correc in the PMS.
