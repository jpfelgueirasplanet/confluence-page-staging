9

BAR Rates in the  OTA_HotelAvailRS Message

  • BAR rates returned in OTA_HotelAvailRS are in line with those in the F11 BAR mode and WBE. The message will display the lowest rate per room type from the BAR group, irrespective of whether it is a virtual or base rate. The message will not display all rates assigned to the BAR rate group. If a rate is required that is not present because it is not the lowest current rate in the BAR Group, then this rate must be moved from the BAR group into a standard group

11

Errors regarding the HTNG_ChargePostingRS

  • The third-party would not be able to process the payment because it took more than 30s for the HTNG_ChargePostingRS to be sent out. In Kibana The HTNG_ChargePostingRS will include a success

 To mitigate the problem the following configuration settings are suggested as may  reduce the comserver load

  1. Make sure PWS is connected to its own taskserver
  2. The new taskserver should have nothing else running against it, no ckits, no IDS
  3. IDS should be pointed to its own taskserver and not the comserver
  4. Taskserver should be set to reload system data cache every 60 minutes
    TaskServer_Setup
  5. Set this on taskserver.ini as that is where the pws is pointing
    [system]
    debugevtimerexecutiontimer=1
    thresholdevtimer=900
  6.  Make sure passerby account is configured and available
  7. Add the xsteup on global level
    ComServer | Whitelist | GetComputerName 
  8. If there is a timeout or error, check what the taskserver displays on the screen.


One way Installation


1

Activate the Trigger and LiveMode for OTA_HotelResNotifRQ & additionally the Global Live Mode.


Message type 

Toggle Trigger

Live Mode

OTA_HotelResNotifRQ

2

Create a test reservation & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs.
  • always provide the reservation number to your integration partner, so that the reservation can afterwards be cancelled 


3

Create and start a bulk upload

  • for reservations (OTA_HotelResNotifRQ) for the last one year and ending with the 31st Dec 2050


BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one.

2

  • Test an incoming Reservation Look Up
    (OTA_ReadRQ)
  • provide the ResID from the test reservation of the One-way Inst. step to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  • Test an incoming Folio request
    (    HTNG_HotelFolioRQ)
  • You can continue with the same ResID 
  • integr. partner sends the HTNG_HotelFolioRQ
  • Check if the RQ consists of the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors
  • Test an incoming posting
    (HTNG_ChargePostingRQ)
  • provide the ResID of the test reservation
  • integ. partner posts an amount on the folio (i.e. -10 Euro)
  • open the folio in the PMS and check if the amount was posted correctly
  • in order to void the posting it needs be posted reversed again
  • Test an incoming Check In (Not supported yet by Neorcha)
    (HTNG_HotelCheckInNotifRQ)
  • provide the ResID of the test reservation (arrival date needs to be current PMS date)
  • check if the room status is on correct status (Checked)
  • integ. partner sends a Check In
  • check the status of the test reservation in the PMS if it is Checked-In 
  • Please make sure that the customer undo the CheckIn in order that the reservation can be cancelled
  • Test an incoming Profile modification
    (OTA_ProfileModifyRQ)
  • Partner sends the OTA_ProfileModifyRQ
  • check if the OTA_ProfileModifyRS has been sent back without any errors and check if the modifications have been processed correct in the PMS 
  • Test an incoming reservation modification (Arrival and departure time will be modified by the integration partner )
    (IO_ResModifyNotifRQ)
  • Partner sends the IO_ResModifyNotifRQ
  • check if the IO_ResModifyNotifRS has been sent back without any errors and check if the modifications of arrival time and departure time have been processed correct in the PMS

Message type

Live Mode

OTA_HotelResNotifRQ

2

Create a test reservation & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs.
  • always provide the reservation number to your integration partner, so that the reservation can afterwards be cancelled 


3

Create and start a bulk upload

  • for reservations (OTA_HotelResNotifRQ) for the last one year and ending with the 31st Dec 2050


BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one.

  • Test an incoming Reservation Look Up
    (OTA_ReadRQ)
  • provide the ResID from the test reservation of the One-way Inst. step to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  • Test an incoming Folio request
    (    HTNG_HotelFolioRQ)
  • You can continue with the same ResID 
  • integr. partner sends the HTNG_HotelFolioRQ
  • Check if the RQ consists of the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors
  • Test an incoming posting
    (HTNG_ChargePostingRQ)
  • provide the ResID of the test reservation
  • integ. partner posts an amount on the folio (i.e. -10 Euro)
  • open the folio in the PMS and check if the amount was posted correctly
  • in order to void the posting it needs be posted reversed again
  • Test an incoming Check In (Not supported yet by Neorcha)
    (HTNG_HotelCheckInNotifRQ)
  • provide the ResID of the test reservation (arrival date needs to be current PMS date)
  • check if the room status is on correct status (Checked)
  • integ. partner sends a Check In
  • check the status of the test reservation in the PMS if it is Checked-In 
  • Please make sure that the customer undo the CheckIn in order that the reservation can be cancelled
  • Test an incoming Profile modification
    (OTA_ProfileModifyRQ)
  • Partner sends the OTA_ProfileModifyRQ
  • check if the OTA_ProfileModifyRS has been sent back without any errors and check if the modifications have been processed correct in the PMS 
  • Test an incoming reservation modification (Arrival and departure time will be modified by the integration partner )
    (IO_ResModifyNotifRQ)
  • Partner sends the IO_ResModifyNotifRQ
  • check if the IO_ResModifyNotifRS has been sent back without any errors and check if the modifications of arrival time and departure time have been processed correct in the PMS
