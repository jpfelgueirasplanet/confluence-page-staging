Set Default Cash Register for incoming PostPayments

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up.

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

1. Define a default cash register in pAir

System data / POS / Payment methods / POS

NG UI:

 

2. Add values that are used to define the default cash register for all IO-related postings in the configuration editor

Cash register definition

  • Add CashRegister and default folder in Configuration/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box

Hint:

Please have in mind that the payment methods in pAir have to be linked to -both- PMS AND CASH - so pAir is capable to interlink the payments to the cashier system you have configured in the step above. In case you refuse to do so, all your payments will be routed by pAir to cashier zero (0) automatically.

Activate Batch Sending
(sum all items of a package into a Batch in the Folio message)

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelFolioRS/BuildBatchTree

  • Create HTNG_HotelFolioRS/BuildBatchTree folder if they do not exist
  • The Content type for HTNG_HotelFolioRS should be null
  • The content type for BuildBatchTree should be Boolean and set to "Yes"

Configure a KSK template for invoices

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelCheckOutNotifRS/TemplateNameInvoice

  • Create HTNG_HotelCheckOutNotifRS/TemplateNameInvoice folder if they do not exist
  • The Content-type for HTNG_HotelCheckOutNotifRS is null
  • The content-type for TemplateNameInvoice is String
  • The Data is the Name of the Template. Important: case sensitive!

  • The Invoice Template needs to be configured in the System Language

One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • Make a modification of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs.
  • Always provide the reservation number to your integration partner.


2

Test  OTA_ProfileCreateRQ/OTA_ProfileModifyRQ 

Code2order is not saving a profile if it is not attached to a reservation. Thus, to test OTA_ProfileCreateRQ/OTA_ProfileModifyRQ 

  • Modify the profile of the test reservation created during the previous OTA_HotelResNotifRQ
  • Create test examples of the individual, company profiles. Verify in Kibana that outgoing OTA_ProfileCreateRQ is answered by RS and no error occurs.
  • Modify the individual company profiles by changing name and address details. Verify in Kibana that the OTA_ProfileModifyRQ is answered by the a OTA_ProfileModifyRS and no errors are present. 
  • Cancel your test reservation afterwards.


3

Create and start bulk upload

  • for reservations (OTA_HotelResNotifRQ) for the next 3 years (starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).
  • Possible to test 2 way during the bulk upload


BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 

4

Data Validation Report - DQE

  • Check that the bulk is successfully finished 
  • The report is to be found in the tab Office (Büro) of the front office in pAir.
    For the DQE only the rooms are being taken into account by Code2Order
  • The DQE is NOT mandatory, only in case of obvious discrepancies a check should be done together with the Code2order Support team.

 

Two-way Installation

Examples

1

Checkin Workflow

1. Test an incoming Profile Creation/Modification → (OTA_ProfileCreateRQ / OTA_ProfileModifyRQ)

 Code2order is not saving a profile if it is not attached to a reservation 

  • Create a test reservation without an allocated room number (arrival date needs to be current PMS date) preferably with 2 guests per room and the profile should hold your email address
    • Code2order will send you the link for your reservation
  • Open the link that code2order sent you via email to be able to test the check-in workflow
  • Create test example profiles (individual & company) in C2O for the created reservation.
    Verify in Kibana that the incoming OTA_ProfileCreateRQ is answered by RS and no error occurs.
  • Amend the profile details in C2O (guest name, address, telephone number...)
  • Verify in Kibana that the incoming OTA_ProfileModifyRQ is answered by OTA_ProfileModifyRS and contains no errors.
  • Check amendments are appearing in the profile in protel.

2. Test an Incoming Add attached guest profile → (IO_ResModifyNotifRQ)  

  • Add an additional guest
  • Verify in Kibana that the incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors.
  • Check the attached guest profile in the pms

3.  Test an incoming Change invoice profile → (IO_ResModifyNotifRQ)  

  • Change the invoice recipient of the previous reservation in C2O
  • Verify in Kibana that incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and no error occurs
  • Check in the PMS if the profile of Folio B is changed

4. Test an incoming invoice detail request  → (HTNG_HotelFolioRQ) 

  • Request the folio of the previous reservation in C2O
  • Verify in Kibana that incoming HTNG_HotelFolioRQ is answered by HTNG_HotelFolioRS and contains no errors.

5. Test an incoming Payment posting → (HTNG_ChargePostingRQ)

  • Post a payment for the previous reservation in C2O via the Dashboard
  • Verify in Kibana that incoming HTNG_ChargePostingRQ is answered by HTNG_ChargePostingRS and contains no errors
  • Check in PMS if the amount was posted correctly

6. Test an incoming availability request → (OTA_HotelAvailRQ) 

  • Request for room availability in C2O
  • Check in Kibana that the incoming OTA_HotelAvailRQ is answered by OTA_HotelAvailRS and contains no errors.

7. Test an incoming check-in →  (HTNG_HotelCheckInNotifRQ)

  • Check whether the room of the test reservation created during OTA_ProfileModifyRQ has the correct status (as configured with XSetUp)
  • Check-in the test reservation in C2O
  • Verify in Kibana that incoming HTNG_HotelCheckInNotifRQ is answered by HTNG_HotelCheckInNotifRS and no error occurs
  • Check in the PMS if the Reservation is checked-in successfully

Void the payment & Cancel the test reservation afterwards

CheckOut Workflow (Departure must be on current day)

1. Test an incoming Profile Modification → (OTA_ProfileModifyRQ)

  • Amend the profile details (guest name, address, telephone number...) of the test reservation that the hotel prepared during the general preparation step
  • The profile should hold the email address of the installer to verify the invoice sent by email
  • Verify in Kibana that the incoming OTA_ProfileModifyRQ) is answered by OTA_ProfileModifyRS and contains no errors.
  • Check amendments are appearing in the profile in protel.

2. Test an incoming invoice detail request  → (HTNG_HotelFolioRQ) 

  • Request the folio of the reservation 
  • Verify in Kibana that incoming HTNG_HotelFolioRQ is answered by HTNG_HotelFolioRS and contains no errors.

3. Test an incoming Payment posting → (HTNG_ChargePostingRQ)

  • Post a payment in C2O
  • Verify in Kibana that incoming HTNG_ChargePostingRQ is answered by HTNG_ChargePostingRS and contains no errors
  • Check if the amount was posted correctly

4. Test an incoming check-out (HTNG_HotelCheckOutNotifRQ)

  • Check if the invoices have no open balance
  • Check-out the reservation in C2O
  • Verify in Kibana that incoming HTNG_HotelCheckOutNotifRQ is answered by HTNG_HotelCheckOutNotifRS and contains no errors.
  • Check in the PMS if the reservation has been checked out
  • Verify the Document data from the HTNG_HotelCheckOutNotifRS and verify if you have received an email with the invoice (check evtl. the SPAM folder).

Void the payment/revenue once finished the test.
