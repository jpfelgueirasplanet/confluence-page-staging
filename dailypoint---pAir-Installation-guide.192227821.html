2-Way Optin x-setting (Marketing Allowed/Not Allowed) 

Explanation

Optin and Optout in pAir is controlled via the marketing checkbox within the guest profile shown below

In order for a third party to be able to alter this checkbox the following settings have to be in place

Allow setting of the checkbox

Marketing and Data processing can be allowed by 3rd party if 

"Configuration/currentProperty/propData/guests/OptInBy3rdParty"is set to true

Allow removal of the checkbox

Marketing and Data processing can be disallowed by 3rd party

"Configuration/currentProperty/propData/guests/OptOutBy3rdParty" is set to true

One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that the outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 

Create test examples of the individual, company, and travel agents profiles and provide the correlation ID's and profile numbers to the integration partner

  • modify the individual, company and travel agent profiles by changing name and address detail. Verify in Kibana that the OTA_ProfileModifyRQ is answered by the a OTA_ProfileModifyRS and no errors are present


2

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    for the last 5 years and ending on 31st December 2099.

  • for profiles (OTA_ProfileCreateRQ) 
    for the last 5 years and to current date.

  • for revenue attached to profiles (OTA_HotelStayInfoNotifRQ) 
    for the last 5 years and to current date.



BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


3

Testing of WrittenConfInst functionality

The WrittenConfInst functionality should be tested for the four-letter types:

Letter

Code

Offer

OFR

Confirmation

CNF

Modification

MOD

Cancellation

CXL

In order to do this, the process is:

  • Create a test reservation in the hotel when there is sufficient availability at the property
  • Retrieve the reservation and within the reservation click on the three dots menu highlighted below to display additional options 

  • Select the option "Confirmation Letter" to display the choices shown below

  • Click on "Open Office" to generate the following screen 

The template field is where the user is able to select which template code they wish to send to dailypoint. The default template type is"Create", which if selected will send the template code "CNF" to dailypoint in the section "WrittenConfInst" of a new OTA_HotelResNotifRQ message.

If the user wishes to send a different template type then clicking on the field template type will display the following options in a drop-down window.

When a template type has been selected then clicking on "Print"  will close the previous screen, and the below dialogue will appear which will, in turn, disappear after a few seconds if no interaction is forthcoming from a user.

This will generate the previously mentioned OTA_HotelResNotifRQ containing the corresponding template type code.


As part of the testing process create an example of each template type and provide dailypoint what the corresponding correlation ID's.

As dailypoint to check the content of these messages and confirm that all is in order!

1-Way installation is complete at this point but please note that the 2-Way cannot begin until dailypoint have fully processed all of the bulk uploads.

Depending on the volume of data this can take them some time and they have advised that a gap of two weeks should be left between the bulk uploads being sent and commence of 2-way testing.


 

Two way Installation

Examples

1

Test an incoming Profile Creation

(OTA_ProfileCreateRQ)

dailypoint supports the creation of three profile types which are:

  • Individual
  • Company
  • Travel Agent 
  • ask the Integration partner to create one example of each of the three supported profile types and provide the CorrelationID and profile number for each
  • Important: check the content of the profile and that the OTA_ProfileCreateRS does not contain any errors


2

Test an incoming Profile Modification

(OTA_ProfileModifyRQ)

dailypoint supports the modification of three profile types which are:

  • Individual
  • Company
  • Travel Agent 
  • request the integration partner amend the three profile types created during the previous OTA_ProfileCreateRQ. They should modify items such as: guest name, address, telephone number, etc (note: salutation is not mapped by dailypoint!).
  • request integration partner provide the CorrelationID for each test and check-in kibana that this present
  • check OTA_ProfileModifyRS contains no errors
  • check amendments are appearing in the profile in protel


3

Test incoming Optin Change (Marketing Allowed/Not allowed)

(OTA_ProfileModifyRQ)

Testing of setting marketing to be allowed

  • Create a profile and verify that the marketing checkbox is not set to be "marketing allowed"
  • Provide dailypoint with the CorrelationID and profile number
  • ask the Integration partner to amend the ShareAllOptOutInd to be "NO" (please note that pAir doesn't support individual sharemarketind i.e. phone, email, address)
  • ask the Integration partner to provide the CorrelationID for the OTA_ProfileModifyRQ and verify its presence in Kibana
  • check that the corresponding OTA_ProfileModifyRS contains no errors
  • check in the OTA_ProfileModifyRQ that ShareMarketAllOptInd = "NO"
  • Verify that the "Marketing allowed" checkbox is activated in the guest profile
  • Repeat the above steps for Company and Travel Agents profiles

Testing of setting marketing to not be allowed

  • Create a profile and verify that the marking checkbox is set to be "marketing allowed"
  • Provide dailypoint with the CorrelationID and profile number
  • ask the Integration partner to amend the ShareAllOptOutInd to be "Yes" 
  • ask the Integration partner to provide the CorrelationID for the OTA_ProfileModifyRQ and verify its presence in Kibana
  • check that the corresponding OTA_ProfileModifyRS contains no errors
  • check in the OTA_ProfileModifyRQ that ShareMarketAllOptInd = "Yes"
  • Verify that the "Marketing allowed" checkbox is not activated in the guest profile
  • Repeat the above steps for Company and Travel Agents profiles
