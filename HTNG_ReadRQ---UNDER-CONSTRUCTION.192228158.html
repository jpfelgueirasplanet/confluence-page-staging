HTNG_ReadRQ

Message Description

Important!

HTNG_ReadRQ is designed to work specifically with the response message HTNG_ResRetrieveRS as a message pair.  OTA_ReadRQ is used with the response message OTA_ProfileReadRS for enumerating matching profiles


The purpose of this document is, to define and describe the messages to be used in protel.I/O to lookup or validate checked in guests via protel.I/O. This functionality can be used by several systems:

  • users at POS systems can lookup guests with different parameters - most likely room number and / or the guests last name. The reservation number of a found guest can be used as referral in a guest transfer payment being part of an invoice mirror transaction

  • SPA and POS systems can validate guests by reading information from an electronic key provided to a guest for his stay- via the UID of the key or the reservation number written onto the key

  • Internet authentication systems can authenticate guests by validating information entered by them on a logon page, they were redirected to in their browser. Acquired information with the logon is a combination of information: reservation number, last name, room number, birth date, etc. depending on the systems possibilities and configuration. The guest can be found with a lookup where some of the information is provided in the lookup request while other - especially various optional logon parameters - are tried to be found in the replied list of guests

The system initiating the request talks to a connected party in protel.I/O called the POS interface. It represents the PMS for the addressed customer. The requesting system addresses the lookup query with customer ID and bearer token of the POS interface.

As usual with protel.I/O, messages are defined to be as close as possible to mechanisms already defined in ”Hospitality Technology Next Generation” (HTNG) for that particular or a similar purpose.

HTNG defines a message called ”HTNG_ReadRQ” already containing almost all mechanisms required to transport a lookup for a checked in guest. In the document ”HTNG_Point_of_Sale_v3.1_2014B_Technical_Specification.pdf” these mechanisms are defined in the category ”Point of Sale System (POS)” in the HTNG Resource ”Point of Sale” on the HTNG Specifications page. It needs to be kept in mind when reviewing the HTNG resources, that the message HTNG ReadRQ is also used for various other purposes. An actually working download link for the document can be found here. The mechanisms used to perform a lookup for a checked in guest are defined in chapters 3.1 and 4.1.


Guest lookup request

In contrast to the HTNG idea to perform multiple lookups with a single request, this is not supported with the POS interface and protel.I/O. The lookup of a guest is done the moment the POS staff requires it. Therefore a single lookup is required, that is why it is sent as a single lookup via protel.I/O. This simplifies the lookup process by using optionally guest or reservation information.

The Hotel to lookup the Reservation is defined in the attribute:

HTNG_ReadRQ/ReadRequests/HotelReadRequest/@HotelCode

The following are the lookup parameters all provided with the ”root” path HTNG_ReadRQ/ReadRequests/HotelReadRequest/Verification and the following attributes and sub tags. In the following enumerations solely the path to the attributes and sub tags underneath this path are shown:

  • GenericLook for checked in guests where this parameter occurs in one of the fields @RoomID, Firstname and SurnameO Reservation
  • @IDLook for checked in guests with the reservation number as provided in this parameterO Room
  • @RoomIDLook for checked in guests on the room number provided in the element /PersonName
  • FirstnameLook for checked in guests with the value from this element as first nameO PersonName 
  • SurnameLook for checked in guests with the value from this parameter as last name/PersonName/@PartialNameIf provided with value ”true”, the values for the parameters Firstname and / or Surname are interpreted as a part of the corresponding field

At least one of the search parameters - (Reservation-)@ID, @RoomID, Firstname, Surname and Generic - is mandatory. If none of these are provided, the reply will be negative. Combining multiple parameters according to information as provided by / about the guest minimises the resulting list and hence reduces the chance for the POS user to choose the wrong guest.
