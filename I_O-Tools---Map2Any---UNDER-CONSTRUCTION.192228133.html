Map2Any Data Transformation

Problem

By default, the messages transferred via the ESB are created and processed based on the HTNG standard. Still, some integrations require a slightly different message dialect or even completely different messages to be communicated with external systems - therefore messages need to be modified in different ways. The needed modifications range from changing the format of one or more field values in the message to rewriting the whole message body into a non-XML formatted object hierarchy (e.g. JSON) transforming the data of some of the provided fields into new fields.

Solution

protel.I/O Map2Any is a data mapping toolchain that can map the data of any incoming message to another data format. This transformation is controlled by XSLT documents and performed by the Enterprise Service Bus during message transfer.

Requirements

To integrate the message transformation toolchain into the development- and processing workflows of protel.I/O, the following basic rules apply:

  • Message mapping should be configurable by the end-user. For the immediate future the Message Mapping will be done only by the IOIntegration team. This will be a chargeable service where we will charge the Integration Partner or Customer on a case by case basis 
  • Mapping rules should be reusable, but only visible to the same team of developers. Therefore, mapping rules should be bound to a developer team.
  • Which transformation should be performed has to be configured on a per-message-basis. Divided by incoming- (custom => HTNG) and outgoing (HTNG => custom) messages.
  • Changing a mapping configuration must not change the behaviour for a service that is deployed on production.
  • One mapping rule should be usable in both directions for incoming and outgoing messages.

Architecture

Data sets

Collection

A collection defines a set of mapping rules that can be applied to the messages defined in an Integration Service. Every collection is owned by a developer team and only the members of this team are able to edit it.

Mapping rule

A mapping rule defines a data mapping between one message and another user-defined message - either XML or JSON. It consists of the following fields or data sets:

Field
Type
Description

collection

string

ID of the collection the rule belongs to (e.g. io.protel.mycollection)

exportXSLT

string

Link to or data of the XSLT for converting an outgoing HTNG message to the external message format

externalMessage

string

Data or identifier that links to the definition of the "external" message format that will be converted from/to (depending on the message flow direction).

The message format is expressed based on the type of the external message. For JSON, this is a JSON-Schema, for XML an XSD.

If this value represents the contents of or a reference to the schema data is an implementation detail.

fieldMappings

FieldMap[]

The fields mapping which connects internal message field with external ones. See the next table for more details.

id

string

ID of the mapping rule. In the full rule ID this is the part after the "/" (e.g. mymappingrule). 
Example for a full rule ID: io.protel.mycollection/mymappingrule

When a rule is specified inside an Integration Service's manifest, the full ID will be used in the mapping field.


Valid values also follow the integration service ID pattern:

Pattern
/^[a-z][0-9a-z_\\-]*$/ 

importXSLT

string

Link to or data of the XSLT for converting an incoming external message to the internal HTNG message type

internalMessage

string

The message type of the HTNG message to map this rule to. 
This is the "internal" side of the transformation, meaning that while the message is inside protel's ESB context, it will have this format.

Field Map

A field map maps a source message field to a field inside the target message format. It also contains conversion instructions if the field value needs to be modified during the mapping.

Mapping fields in a Integration Service's manifest

To be able to execute message transformations, each message definition needs an information, if and which transformation should be performed. For this reason both, incoming- and outgoing message definitions have received a new property named "mapping". This field is a string that contains the full rule ID of the rule to execute during the message transfer.

A mapping rule can only be assigned to a message of the same type as stated in its internalMessage field - mapping it to a different message type would result in an error.

Map2Any UI

The user interface for managing the mapping engine is integrated into the Services Marketplace and consists of two sections: Managing collections and assigning mapping rules to Integration Service's message configurations.

Collection management

Collection list

The collection list shows all collections available to the current user. A search term can be entered to filter the available collections.

Collection editor

The collection editor is used to edit the properties of one selected collection. Each collection is owned by one developer team which has the permission to edit its configuration. 
Also, collections owned by a team can be used in Integration Services owned by this team.

Mapping rule editor

The mapping rule editor is a modal popover that is opened when the user wants to add or edit a mapping rule inside the collection editor.

It has two tabs, one for graphical field mapping and one for editing the XSLTs used for message transformation.

Data Mapping Manager

The Data Mapping Manager is a backend service that provides an API to allow the SMP to configure the data mapping rules stored inside it. It contains all information needed to compile a set of XSLT definitions that are used for message transformation. Additionally, it caches those created XSLTs so that a fast delivery to the protel I/O (ESB) is possible.

MMS

Assigning the ready-to-use mapping rules created in the DMM to incoming and outgoing messages of a service is done in the service's manifest. This involves the MMS whose purpose is the management and storage of manifest data.

The assigment itself is performed on a per-message-basis in the message definition of the manifest. For this purpose both, incoming- and outgoing message definitions contain an additional property called mapping which references the ID of the mapping rule to execute when a message of this type is transferred.

protel I/O message transformation

The transformation handler inside the ESB finally executes message transformations on incoming and outgoing messages based on the transformation rules configured in the service's manifest.

The workflows for both directions are pretty similar. The main differences are which of the two XSLTs is read from the DMM in order to execute the transformation and the location of the transformation process itself in the message flow.

While the transformation for incoming messages is executed right after authorizing the incoming message and before the message is handled by the routing system, an outgoing message passed the routing system until it is right before being transferred to the service's endpoint. Likewise, the import XSLT is used to transform an incoming message into its HTNG representation while the export XSLT will be used for an outgoing HTNG message that is about to be delivered to the external service's backend.
