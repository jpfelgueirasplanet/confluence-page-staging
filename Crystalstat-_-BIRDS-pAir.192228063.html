One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


2

Create and start bulk uploads

  • for reservations
    (OTA_HotelResNotifRQ)

    • Start date: 1st of running month 2 years ago (if today is 1st April 2019 - start date would be 01/04/2017)

    • End date: 01/04/2019

  • for blocks/allotments
    (OTA_HotelInvBlockNotifRQ)     → PWS release >= 2.13.x needed
    • Start date: 1st of running month 2 years ago (if today is 1st April 2019 - start date would be 01/04/2017)
    • End date: 01/04/2019

  • for stay details
    (OTA_HotelStayInfoNotifRQ    )
    • Start date: 1st of running month 2 years ago (if today is 1st April 2019 - start date would be 01/04/2017)
    • End date: 01/04/2019
  • for invoices
    (HTNG_HotelFolioNotifRQ) → PWS release >= 2.14.x     needed
    • Start date: 1st of running month 2 years ago (if today is 1st April 2019 - start date would be 01/04/2017)
    • End date: 01/04/2019


BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 


Use the manual triggers to send out the "bulks" for the following message types:

  • OTA_HotelInvCountNotifRQ 
    → for one year in the future

  • OTA_HotelStatsNotifRQ for PastRoomType (HTNG_PastDailyRoomTypeStatistics_v1)
    → for two years in the past (730 days)
    → could be extended up to five years max. in the past

Data Validation Report - DQE

After the bulk exports for OTA_HotelResNotifRQ  and OTA_HotelStayInfoNotifRQ are completed for one-way, the integration partner will require a DQE report for balancing.

CrystalStats needs two DQE reports

  • one including 'pseudo room revenue'
  • one excluding it

Time frame: 01.04.2017 until 01.04.2021 (2 years past/2 years future - same as bulk uploads)
Type of rev: without VAT (net)
Out of order: without out of order
Pseudo rooms: without 'pseudo room revenue' and one with 'pseudo room revenue'
Format: XLSX

  • after checking if the bulk successfully finished, provide the integration partner the report
  • The Integration partner will process the DQE and send you a report afterwards
  • The report is to be found in the tab Office (Büro) of the front office in pAir.

Widget installation: 


For every new CrystalStat installation there is a widget functionality sold. This means that the clients have the possibility to review their CrystalStat statistics directly in protel Air. 

For installing this, CrystalStat will provide you with a "Widget token". This token just needs to get implemented into the Widget installation in BoB ( see the steps). 


  1. Create an ordinary BoB installation in addition to every CrystalStat main installation and use the manifest "eu.crystalstat.widget"

    So the installation list should look like this:
     

  2. Just insert the widget token given by CrystalStat
  3. Done! The client can find his personal Widget in pAir FO. By clicking on "CrystalStat" the Widget gets displayed in full screen.
