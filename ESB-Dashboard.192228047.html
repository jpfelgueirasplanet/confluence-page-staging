General Paragraphs

In the next paragraph will be described the Support Dashboard, each panel - what was created for. The interval of time for these numbers - is 15 minutes, but the numbers depend on the period of the day, in the screen examples below - there is in the evening 5-6PM. In the graphic presentation - there is indicated the time on the bottom of the panel.

Actions to be taken if something is wrong

ESB - ALL IN, msg
ESB - ALL OUT, msg

These 2 panels show the number of all messages coming IN and OUT

  • if ESB-ALL IN is near 0 value, then the problem is on inbound side - Protel Devops should check Cloudflare (DNS mapping), AWS and internal resources for message routing. It might be also that some internal certificates expired and must be replaced.
  • If ESB-ALL OUT is less than inbound number there could be several issues:
    • Some destination endpoints (mostly those ) might be in invalid state - messages are moved to FPS and retried later
    • ESB-outgoing service is down or in an invalid state. It could be mostly because of AMQ or Redis systems down. Alarms are existing, that are sent to slack and email to warn about such cases.  

ESB - SYNC IN, msg
ESB - SYNC OUT, msg

The next panels displays the number of ESB-Sync IN and OUT messages 

  • If ESB-SYNC IN is 0 or near 0 value for a long time (more than 30mins over the day or more than 1-3h over the night), this means that we have an issue with inbound communication - Protel Devops should check Cloudflare (DNS mapping), AWS and internal resources for message routing. It might be also that some internal certificates expired and must be replaced.
  • if ESB-SYNC OUT has a values less than the IN value, this means that we did not receive a response from remote, or we received it with a higher timeout than we have set for SYNC calls (40sec). Such requests must be investigated and the communication channel must be fixed ASAP. This can be done by investigating Kibana and internal ESB logs. Please inform as soon as possible in the io_esb Slack Channel.

ESB - BULK IN, msg
ESB - BULK OUT, msg

Number of ESB-Bulk IN and OUT messages

Bulk operations are done between one source and one destination and same rules as for SYNC calls are applied. 

ESB - ALL IN/OUT, msg/s

In this panel we can see the number of  IN and OUT messages in a given period of time

If there are gaps in the graph, it means that for that time period we did not have any message processing. This usually means that we have a communication issue which must be fixed ASAP - Protel Devops should check Cloudflare (DNS mapping), AWS and internal resources for message routing. It might be also that some internal certificates expired and must be replaced.

ESB - SYNC IN, msg/s

Number of processed SYNC messages in a given period of time

If there are gaps in the graph, it means that for that time period we did not have any message processing. This usually means that we have a communication issue which must be fixed ASAP - Protel Devops should check Cloudflare (DNS mapping), AWS and internal resources for message routing. It might be also that some internal certificates expired and must be replaced.

ESB - BULK IN, msg/s

Number of incoming BULK messages in a given period of time

It is expected to have empty gaps on this graph, activity on this one is happening only when we receive bulk uploads.

In case a bulk was activated and there is still no activity after then this has to be investigated. It might point to a communication issue between the source of sent messages and ESB.

LB-EXT - ALL IN, req/s

All incoming requests per second which pass through LB-Ext node

If visually there are difference in the image for ESB ALL IN and LB-EXT ALL-IN, this means that some messages does not reach ESB main processing flow. This usually means lack of processing resources on ESB side and must be fixed ASAP by adding more resources or check the internal ESB routing. Please inform as soon as possible in the io_esb Slack Channel.

LB-EXT - SYNC IN, req/s

All incoming SYNC requests per second which pass through LB-Ext node

If visually there are difference in the image for ESB SYNC IN and LB-EXT SYNC IN, this means that some messages does not reach ESB main processing flow. This usually means lack of processing resources on ESB side and must be fixed ASAP by adding more resources or check the internal ESB routing. Error Paragraphs

The next "section" of panels, mostly refers to ERRORS

Name of panel

Description                                                              

Screen

Actions to be taken if something is wrong

ESB - ALL ERRORS, cnt

  1. Number of all ESB errors (both business and application related).
  2. Errors repartition in time for a given time period

  • Check in Kibana for log records and take actions according to their message

1,2. ESB - Large msg errors, cnt

3. ESB - Large msg ok, cnt

  1. Number of ESB large messages errors. 
  2. Graphic presentation large messages with errors.
  3. Graphic presentation of large messages without errors.

  • Check Kibana and contact the 3rd party to which this message is addressed and fix the issue
  • It also might be that there is an AMQ, Redis or Xslt transform issues, in which case Protel Devops must be contacted
  • If message repartition for "ESB - Large msg ok" shows a straight line, this points to an error in the XSLT Transform service and Protel Devops must be contacted to fix it
  1. OXI - CD-Proxy, cnt
  2. OXI - ESB OXI GHA, cnt
  1. Number of CD-Proxy communication errors 
  2. Number of ESB to OXI GHA communication errors (Sync calls)

  • If error persists in time, cd-proxy (part of PCI/DSS) team has to be contacted. Usually issues have one of the following root cause:
    • certificate is expired on cd-proxy side
    • GHA system is down
    • GHA IPs changed
    • cd-proxy IPs changed
  1. OXI - GHA to OXI, cnt
  2. OXI - GHA, cnt
  1. ESB to OXI incoming messages processing error
  2. Number of GHA to OXI (CD-Proxy) communication errors

NOTE: image must be updated to reflect the correct info

LB-EXT - ALL 502, cnt

502 - Bad Gateway

  1. Number of all 502 status errors, which goes through LB-EXT
  2. Graphic presentation of all 502 errors

Any positive value on these panels points to a communication issue. Ideally they must be 0 all the time but due to various network routing issues they might appear from time to time regardless of our actions. If these errors persists, then they point to an system issue and Protel Devops team together with ESB dev team must solve it. Please inform as soon as possible in the io_esb Slack Channel.

LB-EXT - ALL 499, cnt

499 - Client Closed Request

  1. Number of all 499 status errors, which goes through LB-EXT
  2. Graphic presentation of all 499 errors

LB-EXT - ALL 504, cnt

504 - Gateway Timeout

  1. Number of all 504 status errors, which goes through LB-EXT
  2. Graphic presentation of all 504 errors

WS Connections

In this panel are displayed the number of websocket connections. We get the number of connected and disconnected hotels from Redis database.

This panel display a runtime data about connected hotels via websocket interface. If this graphs oscillate or gows down, it usually means an issue on Websocket service. Could be issues with internal certificate, Redis, AMQ, memory or cpu. Dev team must be contacted ASAP to solve these issues if observed
