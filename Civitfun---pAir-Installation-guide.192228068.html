Set Default Cash Register for incoming PostPayments

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up.

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

1. Define a default cash register in pAir

System data / POS / Payment methods / POS

2. Add an xsetup that is used to define the default cash register for all IO-related postings in the configuration editor

Navigate to: "Configuration/currentProperty/propData"

  • Add the value CashRegister
  • Add sub-folder "default" (case sensitive - low case letters !!!) under CashRegister
  • Add value "protelIO"
  • "protelIO" should have the content type: int
  • Set the default cash register number in Data text box



Configure a KSK template for invoices

Navigate to: "Configuration/currentProperty/propData/Interfaces/IO/Gateway"

  • Create HTNG_HotelCheckOutNotifRS and its sub-folder TemplateNameInvoice folder if they do not exist
  • HTNG_HotelCheckOutNotifRS content type is "null"
  • TemplateNameInvoice content type is "string"
  • The Data is the Name of the Template. Important: case sensitive!

  • The Invoice Template must be configured in the "System Language"

Config Editor:

  • Name of the Template:

Note:

  • It is only possible to send the invoice in one language (System language). So you should request the hotel in which language would it prefer the invoice being sent out. 
  • the 'Language' always need to display the System Language but the form itself could hold whatever the hotel wants to send out.

4

Ask the the customer whether specific payments need to be created or if Civitfun should use existing ones. The PMS payment codes needed for postings should be mapped in the dashboard

5

FYI: Before creating test reservations clarify with the property that you will do test reservations with additional ChargePosting in order that this could be canceled afterwards so that the property is informed.

Prepare a test reservation ( departure date should be the installation date and preferably with a room amount of max €1) in advance that would be used to test the check-out functionality during the installation date.


 

Two way Installation

Examples

1

Checkin Workflow 

  1. Test an incoming Reservation LookUp via ResID → (OTA_ReadRQ) 
    • Create a test reservation in PMS
    • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  2. Test an incoming Reservations LookUp via DateRange incl. ResStatus → (OTA_ReadRQ) 
    • Create a test reservation in PMS
    • provide the DateRange and the ResStatus to the Int. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  3. Test an incoming Folio request → (HTNG_HotelFolioRQ)
    • provide the ResID of the test reservation 
    • integr. partner sends the HTNG_HotelFolioRQ
    • Check if the RQ consists the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors.
  4. Test an incoming Check In → (HTNG_HotelCheckInNotifRQ)
    • provide the ResID of the test reservation (arrival date needs to be current PMS date)
    • check if the room status is on correct status (Clean/Checked)
    • integ. partner sends a Check In
    • check the status of the test reservation in the PMS if it is Checked-In 
    • Please make sure that the customer undo the CheckIn in order that the reservation can be cancelled
  5. Test an incoming posting → (HTNG_ChargePostingRQ)
    • provide the ResID of the test reservation
    • integ. partner posts an amount on the folio (i.e. -10 Euro)
    • open the folio in the PMS and check if the amount was posted correctly
    • in order to void the posting, it needs to be posted reversed again

Filter in Kibana for testing:

HotelID:XXXX AND RequestMessageName: HTNG_HotelCheckInNotifRQ AND ModuleName:civitfun.team.civitfunksk

(Replace XXXX with your customer's cloud-id)


Void the payment & Cancel the test reservation afterwards

    • provide the ResID of the test reservation 
    • integr. partner sends the HTNG_HotelFolioRQ
    • Check if the RQ consists the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors.
  3. Test an incoming posting → (HTNG_ChargePostingRQ)
    • provide the ResID of the test reservation
    • integ. partner posts an amount on the folio (i.e. -10 Euro)
    • open the folio in the PMS and check if the amount was posted correctly
    • in order to void the posting, it needs to be posted reversed again
  4. Test an incoming Check Out → (HTNG_HotelCheckOutNotifRQ)
    • provide the ResID of the test reservation 
    • check if the invoices have no open balance
    • integ. partner sends a Check Out
    • check the status of the test reservation in the PMS if it is Checked-Out correctly


Filter in Kibana for testing:

HotelID:XXXX AND RequestMessageName: HTNG_HotelCheckOutNotifRQ AND ModuleName:civitfun.team.civitfunksk

(Replace XXXX with your customer's cloud-id)

Void the payment/revenue once finished the test.
