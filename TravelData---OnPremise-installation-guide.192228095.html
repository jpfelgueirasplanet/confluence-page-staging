One way Installation


1

PWS Configuration

  • enable LiveMode and activate ToggleTrigger for required message types for 1-way connection

Message type

LiveMode

ToggleTrigger

OTA_HotelInvCountNotifRQ

2

Testing Occupancy

(OTA_HotelInvCountNotifRQ)

  • trigger for 1 year the OTA_HotelInvCountNotifRQ once on the server of the customer where the PWS service is running 
    • If today is 01st January 2020 - start date would 01/01/2020
    • End date: 31.12.2020
  • Verify in Kibana that the outgoing messages OTA_HotelInvCountNotifRQ is answered with RS message from the Integration system

Open a browser to post the URLs:
(NOTE - you have to change some details here!)

pattern:
http://IP-address(PWS):Port(PWS)/ota/hotelinvcountnotif/{MPE_hotel}/send/{START:YYYY-MM-DD}/{END:YYYY-MM-DD}

e.g.: http://127.0.0.1:8046/ota/hotelinvcountnotif/1/send/2020-01-13/2020-12-31


WITH PWS BUILD 3.4 AND HIGHER:

http://127.0.0.1:8000/ota/hotelinvcountnotif/{MPE_hotel}/{NUMBER OF DAYS}

 e.g.: http://127.0.0.1:8012/ota/hotelinvcountnotif/3/365

3

Configuring an IDS Instance for I/O: How to configure a protel I/O IDS 

Rate values and Restrictions will be sent through the IDS outbound

  • ESB server connection
  • Add new instance
  • Comserver URL
  • URL for Export
  • SQL database
  • Configure the Email Notification
  • Token and Module Name in the Special Tab
  • Enable "fake" sync mode
  • Activation of PCI Proxy is not required
  • the configuration for the children age range is needed as per example in the screenshot
  • Enable ExportData then Rate values and Restriction values
  • Create DCS mapping (just as requested in the beginning of the installation, ask TravelData and the customer which rate plan codes and room types are needed to be mapped in Protel and then include them in the DCS matchings. Please note that the codes are case sensitive and have to be matched exactly as they are mentioned)

Two way Installation

Examples

1

PWS Configuration

  • enable LiveMode for required message types for 2-way connection

Message Type

LiveMode

OTA_HotelResNotifRS 

OTA_ResRetrieveRS

2

Test an incoming Reservation 

OTA_HotelResNotifRQ

  • ask the integration partner to create a test reservation. Verify if the details have been correctly saved in the PMS.
  • ask the Integration partner to cancel the reservation after the verification is done.


3

Test an incoming Reservation LookUp via ResID
(OTA_ReadRQ) 

  • ask the integration partner to do a reservation lookup using a specific time frame
  • check if the ResRetrieveRS is getting sent back to them with success
Example
<?xml version='1.0' encoding='utf-8'?>
<env:Envelope xmlns:env="http://www.w3.org/2003/05/soap-envelope">
<env:Header>
<p:Action xmlns:p="http://protel.io/soap">OTA_ReadRQ</p:Action>
<p:Source xmlns:p="http://protel.io/soap" Module="backline" ModuleVersion="2020-07-02T10:28:55 (TEST)" Service="io.traveldata.crstd"/>
<p:CorrelationID xmlns:p="http://protel.io/soap">CHECK#1635932039</p:CorrelationID>
<htnga:CorrelationID xmlns:htnga="http://htng.org/PWSWG/2007/02/AsyncHeaders">CHECK#1635932039</htnga:CorrelationID>
</env:Header>
<env:Body>
<OTA_ReadRQ xmlns="http://www.opentravel.org/OTA/2003/05" TimeStamp="2021-11-03T10:33:59+01:00" CorrelationID="CHECK#1635932039" MaxResponses="10">
<POS>
<Source>
<RequestorID ID="14081" Type="10" ID_Context="protelIO"/>
</Source>
</POS>
<ReadRequests>
<HotelReadRequest>
<SelectionCriteria Start="2021-11-03" End="2021-11-07"/>
</HotelReadRequest>
</ReadRequests>
</OTA_ReadRQ>
</env:Body>
</env:Envelope>
