Errors regarding the HTNG_ChargePostingRS

  • The third party would not be able to process the payment because it took more than 30s for the HTNG_ChargePostingRS to be sent out. In Kibana The HTNG_ChargePostingRS will include a success

To mitigate the problem the following configuration settings are suggested as may  reduce the comserver load

  1. Make sure PWS is connected to its own taskserver
  2. The new taskserver should have nothing else running against it, no ckits, no IDS
  3. IDS should be pointed to its own taskserver and not the comserver
  4. Taskserver should be set to reload system data cache every 60 minutes
    TaskServer_Setup
  5. Set this on taskserver.ini as that is where the pws is pointing
    [system]
    debugevtimerexecutiontimer=1
    thresholdevtimer=900
  6.  Make sure passerby account is configured and available
  7. Add the xsteup on global level
    ComServer | Whitelist | GetComputerName 
  8. If there is a timeout or error, check what the taskserver displays on the screen.

General Workflow

  • Reservation set up in protel - Reservation sent to Duve (OTA_HotelResNotifRQ)
  • Pre-Stay E-Mail sent to guest with URL for Duve App/Portal
  • Guest connects to Duve URL


BEFORE CHECK IN

  • Guest changes profile details - Duve -> protel (OTA_ProfileModifyRQ)
  • Guest checks invoice (HTNG_HotelFolioNotifRQ / HTNG_HotelFolioRQ) (Certified, but not yet implemented by Duve)
  • If activated, guest makes deposit payment - Duve -> protel (HTNG_ChargePostingRQ)
  • Guest can purchase add-ons (added as fixed charge) and if activated, make payment (payment to invoice) for these add-ons Duve -> protel (IO_ResModifyRQ / HTNG_ChargePostingRQ)
  • Guest can add arrival and departure time - Duve - protel (IO_ResModifyNotifRQ)
  • Guest can change the room type (upsell), if activated - Duve → protel (Room type change and Revenue Posting as fixed charge via IO_ResModifyRQ)
  • Duve sends information as trace to protel Duve - protel (IO_ResModifyNotifRQ)


CHECK-IN (on day of arrival)

  • Guest can check in via the app - Duve -> protel (HTNG_HotelCheckInNotifRQ)  (Certified, but not yet implemented by Duve)


IN-HOUSE

  • Guest can purchase add-ons and if activated, make payment for these add-ons Duve -> protel (IO_ResModifyRQ / HTNG_ChargePostingRQ)


CHECK-OUT (on day of departure)

  • Guest can check out via the app if the invoice is balanced -  Duve -> protel (HTNG_HotelCheckOutNotifRQ)  (Certified, but not yet implemented by Duve)


3

CheckIn Workflow

1. Test an incoming Profile modification → (OTA_ProfileModifyRQ)
 Duve is not saving a profile if it is not attached to a reservation 

  • Create a test reservation with allocated room number(arrival date needs to be current PMS date) preferably with 2 guests per room and the profile should hold youremail address
    • Request Duve to trigger the email with the link for your reservation
  • Open the link that Duve sent you via email to be able to test the check-in workflow
  • Amend the profile details in Duve (go through the quick check in procedure) until signature.
  • Verify in Kibana that the incoming OTA_ProfileModifyRQ is answered by OTA_ProfileModifyRS and contains no errors.
  • Check amendments are appearing in the profile in protel.

 2. Test incoming Trace → (IO_ResModifyNotifRQ)

  • Check that the trace has been update in the reservation in protel
  • Verify in Kibana that incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors

3. Test incoming arrival time → (IO_ResModifyNotifRQ)

  • Check that the arrival time has been update in protel
  • Verify in Kibana that incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors

Reopen the app in the browser

4. Test purchase add-on →(IO_ResModifyNotifRQ)

  • Purchase an add-on in the app (without payment)
  • Verify in Kibana that incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors
  • Check in PMS that the fixed charge is set up correctly.

4. Test an incoming invoice detail request → (HTNG_HotelFolioRQ)  NOT IMPLEMENTED YET BY DUVE

  • Request the folio of the previous reservation in Duve
  • Verify in Kibana that incoming HTNG_HotelFolioRQ is answered by HTNG_HotelFolioRS and contains no errors.

5. Test an incoming Payment posting → (HTNG_ChargePostingRQ) (if payment activated)

  • Post a payment for the previous reservation in Duve App
  • Verify in Kibana that incoming HTNG_ChargePostingRQ is answered by HTNG_ChargePostingRS and contains no errors
  • Check in PMS if the amount was posted correctly

5. Test room upgrade, if activated → (IO_ResModifyNotifRQ)

  • Choose an upgrade room type and confirm
  • Verify in Kibana that incoming IO_ResModifyNotifRQ is answered by IO_ResModifyNotifRS and contains no errors
  • Check the room type has been changed and check that the upgrade has been added correctly as fix charge in the PMS

6.  Test an incoming Check-In →  (HTNG_HotelCheckInNotifRQ) NOT IMPLEMENTED YET BY DUVE

  • Check whether the room of the test reservation created during OTA_ProfileModifyRQ has the correct status (as configured with XSetUp)
  • Check-in the test reservation in Duve
  • Verify in Kibana that incoming HTNG_HotelCheckInNotifRQ is answered by HTNG_HotelCheckInNotifRS and no error occurs
  • Check in the PMS if the Reservation is checked-in successfully

Void the payment & Cancel the test reservation afterwards
