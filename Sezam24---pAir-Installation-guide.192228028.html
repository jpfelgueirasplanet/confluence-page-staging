Actions at least one day BEFORE the installation day

Please make sure with the property that there is a prepared test reservation with departure for the current date in order to be able to test the CheckOut.

As it is NOT possible to test the essential KIOSK functionalities such as Check-In / Check-Out and Walk-In remotely, which means these tests require trained staff on-site that can do the tests together with protel and under supervision of the integration partner, assure BEFORE starting with the installation that the KIOSK system is on-site, at least one person is available on the testing day that can do the tests with us. Otherwise, as it is not possible to test key functionalities cancel the installation and assure the prerequisites are given.

Verify with the customer which KIOSK functionalities are licensed and used by the customer - not all functionalities are enabled by the standard. The customer's feature using is optional: Walk-In // Check-In // Check-Out

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up. 

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

  1. Define a default cash register in pAir

System data / POS / Payment methods / POS

2. Add an xsetup that is used to define the default cash register for all IO-related postings in the configuration editor

  • Add CashRegister and default folder in Configuration-editor: root/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box

Hint:

Please have in mind that the payment methods in pAir have to be linked to -both- PMS AND CASH - so pAir is capable to interlink the payments to the cashier system you have configured in the step above. In case you refuse to do so, all your payments will be routed by pAir to cashier zero (0) automatically.

One way Installation

1

Create a test reservation in the future (at least one year after the system date)

  • make a modification and finally cancellation of your test reservation.
    Verify-in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 
  • always provide the reservation number to your integration partner, so that the reservation can afterward be canceled

2

Make a modification on the invoice (i.e. post a revenue/payment on the folio) of the initially created test reservation (day before)

  • In case there is already a posting on the invoice you may ask the vendor to read it and pass the information to you.
  • Verify-in Kibana that outgoing HTNG_HotelFolioNotifRQ is answered by HTNG_HotelFolioNotifRS and no error occurs.
  • make sure that the test posting on the invoice gets voided once the test is done, otherwise, there will be revenue discrepancies.

Kiosk Workflow overview 

Steps for WalkIn workflow:

  1. IO_SystemStatusRQ to Protel
  2. OTA_HotelAvailRQ to Protel in order to get the RoomAvailabilities
  3. OTA_HotelResNotifRQ to Protel with an empty profile (Protel responds with created ProfileID)
  4. OTA_ProfileModifyRQ to Protel with full Profile
  5. OTA_ReadRQ to Protel (via ResID or DateRange)
  6. HTNG_HotelFolioRQ to Protel
  7. HTNG_HotelCheckInNotifRQ to Protel
  8. HTNG_ChargePostingRQ to Protel

Steps for CheckIn workflow:

  1. IO_SystemStatusRQ to Protel
  2. OTA_ReadRQ to Protel (via ResID or DateRange)
  3. HTNG_HotelFolioRQ to Protel
  4. HTNG_HotelCheckInNotifRQ to Protel
  5. HTNG_ChargePostingRQ to Protel

EOD

Steps for CheckOut workflow:

  1. IO_SystemStatusRQ
  2. OTA_ReadRQ to Protel (via ResID or DateRange)
  3. HTNG_HotelFolioRQ to Protel
  4. HTNG_ChargePostingRQ to Protel (invoice balance =0)
  5. HTNG_HotelCheckOutNotifRQ to Protel

FYI: Before creating test reservations clarify with the property that you will do test reservations with additional ChargePosting in order that this could be canceled afterward so that the property is informed.

WalkIn workflow (!! If WalkIn is needed start here and you can skip check-in workflow !!)

  1. Test an incoming SystemDate lookup → (IO_SystemStatusRQ)
    • Partner sends the IO_SystemStatusRQ
    • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  2. Test an incoming Availability check → (OTA_HotelAvailRQ)
    • Provide the Timespan for which the available rooms will be requested
    • Please make sure that rooms are vacant (at least one room)
    • integration partner sends in OTA_HotelAvailRQ
    • make spot checks on rooms in OTA_HotelAvailRS and PMS if they coincide
  3. Test an incoming Reservation  → (OTA_HotelResNotifRQ)
    • Please make sure that rooms (at least one room) according to room type is vacant and clean/checked upon arrival date
  4. Test an incoming Reservation LookUp via ResID and/or DateRange incl. ResStatus→ (OTA_ReadRQ) 
    • provide the ResID to the integration partner in order to see if the ResRetrieveRS is getting sent back to them with success
  5. Test an incoming Folio request → (HTNG_HotelFolioRQ)
    • provide the ResID of the test reservation 
    • The integration partner sends the HTNG_HotelFolioRQ
    • Check if the RQ consists of the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors
  6. Test an incoming Check-In → (HTNG_HotelCheckInNotifRQ)
    • provide the ResID of the test reservation (arrival date needs to be current PMS date)
    • check if the room status is on correct status (Clean/Checked)
    • The integration partner sends a Check-In
    • check the status of the test reservation in the PMS if it is Checked-In 
    • Please make sure that the customer undo the CheckIn in order that the reservation can be canceled
  7. Test an incoming posting → (HTNG_ChargePostingRQ)
    • provide the ResID of the test reservation
    • integration partner posts an amount on the folio (i.e. -10 Euro)
    • open the folio in the PMS and check if the amount was posted correctly
    • in order to void the posting, it needs to be posted reversed again

Checkin Workflow 

  1. Test an incoming SystemDate lookup → (IO_SystemStatusRQ)
    • Partner sends the IO_SystemStatusRQ
    • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  2. Test an incoming Reservation LookUp via ResID → (OTA_ReadRQ) 
    • Create a test reservation in PMS
    • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  3. Test an incoming Reservations LookUp via DateRange incl. ResStatus → (OTA_ReadRQ) 
    • Create a test reservation in PMS
    • provide the DateRange and the ResStatus to the Int. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  4. Test an incoming Folio request → (HTNG_HotelFolioRQ)
    • provide the ResID of the test reservation 
    • integr. partner sends the HTNG_HotelFolioRQ
    • Check if the RQ consists the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors.
  5. Test an incoming Check In → (HTNG_HotelCheckInNotifRQ)
    • provide the ResID of the test reservation (arrival date needs to be current PMS date)
    • check if the room status is on correct status (Clean/Checked)
    • integ. partner sends a Check In
    • check the status of the test reservation in the PMS if it is Checked-In 
    • Please make sure that the customer undo the CheckIn in order that the reservation can be cancelled
  6. Test an incoming posting → (HTNG_ChargePostingRQ)
    • provide the ResID of the test reservation
    • integ. partner posts an amount on the folio (i.e. -10 Euro)
    • open the folio in the PMS and check if the amount was posted correctly
    • in order to void the posting, it needs to be posted reversed again

CheckOut workflow (Departure must be on current day)

  1. Test an incoming SystemDate look up → (IO_SystemStatusRQ)
    • Partner sends the IO_SystemStatusRQ
    • check if the IO_SystemStatusRS has the current PMS date included (Check "CurrentBookingDate")
  2. Test an incoming Reservation LookUp via ResID → (OTA_ReadRQ) 
    • provide the ResID to the integ. partner in order to see if the ResRetrieveRS is getting sent back to them with success
  3. Test an incoming Folio request → (HTNG_HotelFolioRQ)
    • provide the ResID of the test reservation 
    • integr. partner sends the HTNG_HotelFolioRQ
    • Check if the RQ consists the correct ResID and if the HTNG_HotelFolioRS has been sent back without any errors.
  4. Test an incoming posting → (HTNG_ChargePostingRQ)
    • provide the ResID of the test reservation
    • integ. partner posts an amount on the folio (i.e. -10 Euro)
    • open the folio in the PMS and check if the amount was posted correctly
    • in order to void the posting, it needs to be posted reversed again
  5. Test an incoming Check Out → (HTNG_HotelCheckOutNotifRQ)
    • provide the ResID of the test reservation 
    • check if the invoices have no open balance
    • integ. partner sends a Check Out
    • check the status of the test reservation in the PMS if it is Checked-Out correctly
