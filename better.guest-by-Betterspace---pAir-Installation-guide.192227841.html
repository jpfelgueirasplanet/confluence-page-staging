Installation Steps

 

General steps

Examples

1

Request authentication details from the Integration partner for the new installation and request them to send you the authentication details on private Slack message or encrypted file by email and inform you by phone about the password.

 BetterSpace are using WSSecurity user name and password. Those are set per protel customer. The EP is pre-set as default.


Please provide them the HotelID as well - internally for Protel this is the CloudID 

2

Provide to the integration partner the list of active payment methods from the SD of the customer. BetterSpace can use only one type of CC payment - if necessary, special payment method could be created by the customer for the purpose.

Additionally, the customer should create a fallback payment method just in case a payment method which is used, does not exist 

Confirm with the partner if they have completed the communication and setup with the customer in terms of:

  • Configuration for payment provider complete (credentials, setup)
  • Clear room bill only during C/O or also during C/I.

3

Communicate with the Customer:

  • which cash register ID should be used - or if new one needs to be created. 
  • which should be the default invoice template and add this one in the PWS config.
  • which payment method should be used - or if new one needs to be created
  • Ask the customer to prepare a test reservation that can be checked out on the day of the installation with minimum amount due.


4

General Preparation for pAir General (pAir)



5

Special XSetup Configuration

Set Default Cash Register for incoming PostPayments

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up.

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

Define a default cash register in pAir
Settings / Financial Configuration / Cash Registers

Add an xsetup that is used to define the default cash register for all IO-related postings in the configuration editor

  • Add CashRegister and default folder in Configuration-editor: root/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box

Set the desired payment method:

Activate Batch Sending
(sum all items of a package into a Batch in the Folio message)

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelFolioRS/BuildBatchTree

  • Create HTNG_HotelFolioRS/BuildBatchTree folder if they do not exist
  • The Content type for HTNG_HotelFolioRS should be null
  • The content type for BuildBatchTree should be Boolean and set to "Yes"


Configure a KSK template for invoices

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelCheckOutNotifRS/TemplateNameInvoice

  • Create HTNG_HotelCheckOutNotifRS/TemplateNameInvoice folder if they do not exist
  • The Content type for HTNG_HotelCheckOutNotifRS is null
  • The content type for TemplateNameInvoice is String
  • The Data is the Name of the Template. Important: case sensitive!

  • The Invoice Template needs to be configured in the System Language

  • It is only possible to send the invoice in one language (System language). So you should request the hotel in which language would it prefer the invoice being sent out. 
    • the 'Language' always need to display the System Language but the form itself could hold whatever the hotel wants to send out.
  • Hotelbird will forward the invoice to the guest and the email will be sent depending on the guest language

One way Installation


1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 


2

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    from today and ending with the next 2 weeks

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 

 DQE is not needed - instead - confirm if the total number of unique CIDs sent by the PMS = the total number of such received by the partner

This can be achieved by using the Kibana filters:

  • LogMessage: "New bulk request coming in"
  • RequestMessageName: "OTA_HotelResNotifRQ"
  • HotelID: "XXXX", where the "XXXX" is the respective customer cloud ID

Which should also match with the SMP record from the bulk:

 

Two way Installation

Examples

1

Test an incoming Check-in flow

OTA_ReadRQ → OTA_HotelAvailRQ → IO_ResModifyNotifRQ → OTA_ProfileModifyRQ → HTNG_HotelFolioRQ → HTNG_HotelCheckInNotifRQ

  1. Create a test reservation (arrival date needs to be current PMS date) and provide the reservation number to your integration partner
  2. Check whether there are available room/s to assign with the correct status "clean" or "checked" (depending on customer preference)
  3. Provide the reservation number of the test reservation during the general preparation step
  4. Ask the integration partner to check-in the test reservation
  5. Verify in Kibana the RQ message from partner:
    • OTA_ReadRQ - is responded with OTA_ResRetrieveRS
    • HTNG_HotelFolioRQ - is responded with HTNG_HotelFolioRS
    • OTA_ProfileModifyRQ - is responded with OTA_ProfileModifyRS. Check in the PMS if the profile has changed
    • OTA_HotelAvailRQ si responded with OTA_HotelAvailRS
    • IO_ResModifyNotifRQ is responded with IO_ResModifyNotifRS
    • HTNG_HotelCheckInNotifRQ is responded with HTNG_HotelCheckInNotifRS. Check in the PMS if the reservation was checked-in successfully to the correct room
  6. Make sure that none of the API's flows above have an error logged.
  7. Let the integration partner know that you will cancel the reservation. Undo the check-in and cancel the test reservation afterwards
