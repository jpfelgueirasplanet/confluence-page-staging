Before a two-way RMS integration using protel I/O can be activated for a hotel, the hotel's rates will need to be prepared accordingly.

This preparation includes:

  • setting up protel Daily Rates for SPE/MPE properties evaluating
  • adjusting the rate structure for the future application with an RMS


This document outlines the steps that are necessary for the hotel to efficiently apply an RMS.
Use it to ensure that the rates supplied by the future RMS are correctly represented and processed by their protel PMS, including the optional export to an IDS instance or WBE.


Updating and/or setting up protel Daily Rates

Initially, check out if you are already using protel Daily Rates or if you are still using the old Room Rate Adjustment (RRA).

Why protel Daily Rates?

Several times a day, the RMS transmits (pushes) rates to the protel PMS per rate code, room type and occupancy to the protel PMS.
Using protel CKit Daily Rates ensures that the system data is not overwritten during each of these pushes.
The current version of Daily Rates (version 2.0, or later) manages combinations of rate code and room type plus occupancy.

Side effect: By means of the Daily Rates user interface, rates are also visible to Front Office users (if the user rights are set respectively).

If you are not yet using protel Daily Rates, please get the latest version (version 2.0, or later) installed.
If Daily Rates is already being used, please update to version 2.0 or later.

Status analysis: Rate strategy "prior to RMS"

To prepare the rate adjustments that are going to be made later, check how your current rate structure is made up. This will give you a first impression about whether or not it can be displayed in the RMS without any problems.

  • What strategy underlies the client use for the current rate setup?
  • What is going to change when an RMS is connected to their PMS?
  • Can the rates be completely managed in RMS or will some of them continue to be managed in protel?
  • Does the client use virtual rates? With special rates? Packages? Rate groups?
  • How are the rates currently created?

protel IDS connect or WBE?

While the rates are being adjusted, rate exports can/will possibly be delayed. Before the rate setup starts, please check to answer the following questions:

  • Are you using protel IDS Connect?
  • Are you using a WBE?
  • Which rates are being used by the WBE or IDS instance?

Old or new rates?

The new or extended rate structure should make it easier to create rates and to manage them later.
Therefore, it is important to carefully go over the following points before you decide on extending existing rates or creating new ones:

  • For a clean transition, we recommend creating new rate codes including rate details to be used by the RMS.
  • Please note: All existing reservations are altered, if the details of previous rates will be changed. This must be avoided!
  • If existing rates are going to be used, it is absolutely necessary that new rate details are added (see directions below)!
  • Please also emphasize that while working on the rate code details, all of the protel IDS instances using the respective rates must be deactivated ("Sync only").

Setting up rate code details

Before you start creating new rate code details for old or new rates, please pay attention to the following points:

Considering the appropriate time for the rate adjustment

Of course it is possible to create the rates on the day of the switch over. However, we recommend you take the time to prepare the new rate code details and to first deactivate the old rates on the day of implementation.
Even if the rates need to be manually entered into protel Daily Rates until the RMS begins pushing the data, this procedure is much more relaxed and less prone to errors.

  • Before starting with the rate adjustment, check to see if the RMS is pushing default rates or only changes. In this case, to avoid the rate being left empty, a default value will need to be set in Daily Rates for each rate code and/or room type for at least the next 360 days. These default values will then be overwritten for the respective combination during a push.
  • Please check as well in advance with the RMS system if they could push only one rate code or eventually several ones.
  • As the RMS pushes the actual rates to Daily Rates, there must not be any values entered into the rate code details.
  • The rate code details need to be linked to the Daily Rates split table. Further split tables for breakfast, etc. can be added. — When adding new split tables, take note if the split in question is inclusive or exclusive!
  • Please make sure that the Daily Rates split table is assigned first, otherwise inclusive charges might not be split because the base value in the rate code detail is 0.00.
  • If you would like to work with virtual rate codes based on the rate code the RMS is sending please see the examples below. Furthermore Support team needs to add a configuration to the ProtelWebService which multiplies the entries in the Daily Rate module for each virtual rate.

Example

  1. Base Rate which is filled by the RMS with rate values via Daily Rates module
    1. No special settings for the rate code itself
    2. Create one or several rate code details depending on if you have specific details which change e.g. per room type / season like package assignments or occupancy.



    3. Assign (at least) the Daily Rate split table to these details

  2. Virtual Rate which should have the same rate values but has different packages assigned (e.g. breakfast)
    1. Some special settings for the rate code as it is a virtual rate
      1. Base Rate: BAR
      2. Rate should use its own split tables
    2. Create one or several rate code details depending on if you have specific details which change e.g. per room type / season like package assignments or occupancy.



    3. Assign the Daily Rate split table and all further splits/packages to these details which should be valid for this rate



  3. An advance purchase rate that is 10% off the Base Rate but has breakfast included.
    1. Some special settings for the rate code as it is a virtual rate
      1. Base Rate: BAR
      2. Rate should use its own split tables
      3. Surcharge -10%
      4. Percentage should be applied to split tables as well
    2. Create one or several rate code details depending on if you have specific details which change e.g. per room type / season like package assignments or occupancy.



    3. Assign the Daily Rate split table and all further splits/packages to these details which should be valid for this rate. All splits in further packages which should NOT be discounted by 10% need to be marked accordingly


Now if the RMS sends a rate of 100.00 it would display as follows in F11 (reservation enquiry screen)


  • for 1 adult:
    • BAR rate € 100.00
    • BARBB rate € 115.00
    • ADVBB rate € 105.00

  • for 2 adults it would be:
    • BAR € 100.00
    • BARBB € 130.00
    • ADVBB € 120.00
