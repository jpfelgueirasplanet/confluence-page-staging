<< General Approach when trouble shooting with Kibana log message >>

  1. Read the error message received - some of those point unambiguously to the reason/troubleshooting requirements. If that does not help, then:
  2. Analyse the entire workflow of the issue and the way it is logged in Kibana - it should match to one of the expected flows defined below. Looking at which point the error appears (or the flow stops) might give guidance where to continue the search. If that does not help, then:
  3. Check initial configurations (API communication settings), e.g.:
    1. BOB installation 
    2. BOB manifest
    3. PWS IO Configuration
    4. IDS settings
    5. Handover documentation specifics (if for PROD env)
      (the below will be relevant to TEST env)
  4. Try to replicate the scenario using Postman
  5. Ask the partner to try 1-4 on their end

 Some error messages are connected/related. It is important to analyze the entire flow. The message you're trying to solve might only be a consequence of the root cause, explained in a previously occurring error message.

 If none of the above helps - refer to the table with Troubleshooting ideas below. If the description there still does not help - contact #io_integrations or #io_esb teams (or #io_partners)

 If the error affects a lot of (all) customers - it is most probable that it is on protel end

<< Minimum information needed when escalating an issue >>

  1. Environment
  2. Hotel ID/Cloud ID
  3. CID (at least 1 example)
  4. Error message
  5. Steps to re-produce
  6. Describe the expected and the actual result

ESB Outbound

N

Kibana LogMessage

Description / Meaning

Trouble Shooting / Actions Required


E1

Moved to Fault queue based on dead endpoint marker

This type of error indicates a dead endpoint from the remote. The affected vendor endpoint will be set as dead for 20mins then the dead EP marker will be deleted allowing all messages received by ESB in meantime to be sent out, or resent if got timeout on initial attempt.

No action required. 

If error persists in time, remote vendor must be contacted to inform that their system is unstable and ask to fix it.


E2

Moved to Fault queue based on lazy client detector

This type points to a special type of clients, lazy vendors which are known to be unresponsive (very long ACK or always timeout from their side). When one such a client is added (manually in ESB config, not BOB) any request to be delivered out to this vendor will automatically sent out to FPS service for retries instead of trying it on ESB flow.

This might be useful when we observe vendors that may affect the ESB flow by consuming critical resources (waiting 10-15sec per request is very expensive) and they don't fix their system in due time but in the same time we are forced by the signed contract with them to deliver any messages for their destination.

These messages will be delivered to the destination, just that they will be delivered using FPS strategy with incremental redelivery delay

  1. We must ask the vendor to improve their system performance
  2. If not possible, to move them to throttling flow


E3

SOAP11 fault processing not supported yet

Means that for now, any failed to be delivered SOAP11 request will not be queued for retries. We may add support for it in a future ESB release version if needed

No action required


E4

Cannot extract authorization key for given hotel id and service name

When we detect a case when ESB cannot deliver a message out to a vendor due to some checked restrictions or invalid response received from remote or in case of an ASYNC/SYNC flow we must deliver back to ESB as loopback the final response to finalyze the transaction request. For this, we are using the hotel id and destination service to identify the token used by the destination vendor to send final response back to ESB.

In case the token cannot be identified via this combination (hotel + destination service), an error is logging in kibana and message processing stops.

This points to some misconfiguration in ESB config received from BOB. Try to disable/enable the installation which will trigger a fresh config version to be delivered to ESB.

If error persists, open a JIRA ticket for ESB team


E5

Failed to send message out

Logged when a HTTP 4xx was received when trying to send a request out to a remove vendor and a final RS is sent back to ESB as loopback to inform about this error (otherwise sender will never receive a final RS from remote to know if there was a successful or invalid processing for their message)

Investigate the message payload for possible invalid data. Ask remote vendor to describe the error and if possible to provide the possible solution from our side (if any)


E6

Reached max number of retries - move to ActiveMQ.DLQ

Logged for the case when message reached maximum number of retries or maximum processing time for failed requests (24h) reached

No action required

If error persists in time, remote vendor must be contacted to inform that their system is unstable and ask to fix it.


E7

Drop RS message since not possible to send back a RS for an incoming RS message

This message is logged in case of a circular message check and this message type is RS. We cannot send back an RS message to the sender as they dont expect any further action once the RS message was sent out to ESB.

No action required


E8

Dropped due to circular message flow detection

This message is logged when ESB detected a circular message (CausedBy element in Soap Header points to an previous transaction with the same CID)

No action required


E9

Failed to build and send back circular message response

Logged when we failed to send back final RS as loopback to ESB to inform about the circular message detection. It usually happens when ESB-in is in an unstable state (large load, bugs after PROD deploy)

If error persists, open a JIRA ticket for ESB team


E10

Message not allowed to be sent out to websocket service

Such a messages is acceptable for HotelID with multiple installations. The problem affects all customers with IDS installation - so the more such are available, the higher the queue with open WS connections that can not move forward in ESB.

Currently - this message shows that a redundant message was dropped from being directed towards an existing IDS installation for the customer. You should treat this message as "expected"/"normal" workflow

No action required

Remote vendor must be contacted to fix this flow


E26

Failed to process message - cannot finish ASYNC/SYNC flow due to invalid loopback response from ESB

Sender sends ASYNC RQ and expects only RS back but receiver provided an ACK instead of the actual RS. Thus, it is considered that the RQ msg was successfully delivered - only the workflow is broken. To avoid duplicate postings - the message will not be retried. 

Check with the recipient (service destination) of the failed msg for the exact failure reason.
Analyze the workflow itself - e.g. ACK (for async) / RS for sync within acceptable time frames.

E1

Moved to Fault queue based on dead endpoint marker

This type of error indicates that remote target did not respond for any of the messages sent out to it and reached a threshold of 50 timed out (>15sec) messages for 1min. 

All messages that goes for this destination once it is declared dead, will be automatically moved to FPS service queue for next 20 mins starting from the moment when dead marker is set

  1. There is no action required for removing the Dead EP - this is automatically done in 20 minutes
  2. No messages are lost - all messages triggered while the Dead EP flag is raised are being stored - simply "delayed"
  3. The very action needed is to investigate the cause for putting the partner EP to the "Dead" flag - check the error messages logged exactly before the dead EP marker was raised

"http://www.w3.org/2003/05/soap-envelope"
