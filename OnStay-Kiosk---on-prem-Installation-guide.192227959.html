Errors regarding the HTNG_ChargePostingRS

  • The third-party would not be able to process the payment because it took more than 30s for the HTNG_ChargePostingRS to be sent out. In Kibana The HTNG_ChargePostingRS will include a success



To mitigate the problem the following configuration settings are suggested as the may reduce the comserver load

  1. Make sure PWS is connected to its own taskserver
  2. The new taskserver should have nothing else running against it, no ckits, no IDS
  3. IDS (all configured instances) should be pointed to its own taskserver and not the comserver 
  4. Taskserver should be set to reload system data cache every 60 minutes
    https://wiki.protel.net/index.php?title=TaskServer_Setup
  5. Set this on taskserver.ini as that is where the pws is pointing
    [system]
    debugevtimerexecutiontimer=1
    thresholdevtimer=900
  6.  Make sure passerby account is configured and available
  7. Add the xsetup on global level
    ComServer | Whitelist | GetComputerName 
  8. If there is a timeout or error, check what the taskserver displays on the screen.

2

Test a Check In 

Check In Flow:

  • Kiosk searches for the guest reservation 
    OTA_ReadRQ
    HTNG_HotelFolioRQ
  • Reservation Info is shown to the guest
  • Guest is shown profile information
  • Guest completes profile information
  • Window appears asking if an email invoice is required and the possibility to enter an invoice address

This information is NOT pushed back to the PMS.  An email is sent to the hotel with the information and has to handled manually by the hotel staff!

  • Guest chooses how many key cards are required
  • Profile information is pushed to PMS
    OTA_ProfileModifyRQ
  • Guest makes payment at the Kiosk  (Invoices with Payment Method set to "Accounts Receivable" are not considered and not shown on the Kiosk)
  • Key cards are created
  • Payment is posted to PMS
    HTNG_ChargePostingRQ
  • Check In is completed
    HTNG_HotelCheckInNotifRQ
  • To test the check in flow, create a reservation in the PMS for the actual business date and provide the reservation ID, Last name and firstname to OnStay.


3

Test a Walk In

Walk In Flow:

  • Guest checks for availability on the Kiosk
    OTA_HotelAvailRQ

    There must be free and clean rooms available, otherwise the kiosk will show no availability.  


  • Guest can choose room type
  • Guest is shown profile information
  • Guest completes profile information and signs
  • Summary of booking is shown
  • Profile information and reservation is pushed to PMS
    OTA_ProfileModifyRQ
    OTA_HotelResNotifRQ
  • Reservation Info is shown to the guest
  • Key cards are created
  • Guest makes payment at the Kiosk
  • Payment is posted to PMS
    HTNG_ChargePostingRQ
  • Check In is completed
    HTNG_HotelCheckInNotifRQ
  • To test the walk In flow, as the partner or Customer to create a new booking via the Buchen or Booking button on the kiosk and complete the check in immediately.

4

Check Out

A Check Out option is available on the kiosk. This option only sends the Check Out Message to the PMS.  It does not capture the invoice document and send it to the guest.  
Invoice balance must be 0.00, otherwise the Check Out will fail.  Payment will not be processed in the Check Out process.

  • Guest searches for reservation
    OTA_ReadRQ
  • Reservation shown in the Kiosk
  • Guest confirms check out
  • Check Out is done
    HTNG_HotelCheckOutNotifRQ
  • ask the Integration partner to clarify with the customer to have the Test rate code invalidated (through expired validity) after the verification of the Restriction message is done
  • ask the Integration Part in the Slack channel of the integration to push a restriction for a specific day on the Test rate code and then check in the Price availability for the given date if the restriction has been set.
