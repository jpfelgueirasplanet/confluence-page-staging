Data Security

Please have in mind that both vendor and IO interface token information is sensitive information that might be transported by Slack,
team viewer session in case of a 1:1 session, or with two-way authentication methods by other media. 

  • the domain of the EP per hotel need to be requested with no Authentication Method
  • The EP's domain are on every hotel different (the XXXX of the pattern bellow). The "path" ( eg /protelIO.rateplans.import) is different per xml message but the same on every hotel. Bellow there is is a pattern:
    RQ to Interalp (outbound)
    OTA_HotelInvCountNotifRQ (API_INVURL) with Value to https://XXXX/protelIO.availabilities.import
    OTA_HotelRatePlanNotifRQ (    API_RATEURL) with Value to https://XXXX/protelIO.rateplans.import
    OTA_HotelAvailNotifRQ  (API_RESTRICTIONURL) with Value to https://XXXX/protelIO.rules.import

    all the RS back to Interalp ( from inbound RQ)
    OTA_HotelResNotifRS, OTA_HotelAvailNotifRS, OTA_HotelRatePlanNotifRS, IO_SystemDataRS (API_RESPONSEURL) with Value to https://XXXX/protelIO.response.import
  • Reservation Target (Res Target) should be on protel Air

Configuring an IDS Instance for I/O for rate and restriction export: 

pAir | How to configure a protel I/O IDS

if Rates and Restrictions export is not a applicable for an installation then IDS installation is not necessary

  • Rate values and Restriction Values are necessary to be ticked in order to be exported
  • Please note that availability Values are not needed, so no tick is necessary!
  • Ask Interalp and the customer which rate plan codes and room types are needed to be mapped in Protel and then include them in the Mappings. Please note that the codes are case sensitive and have to be matched exactly as they are mentioned!

Two way Installation

Examples

1

Test an incoming Reservation 

(OTA_HotelResNotifRQ)

  • ask the Integration partner to cancel the reservation after the verification is done.
  • Important: check if the integration needs reservations with Credit Card, and create a test reservation with CC details. Verify if the details have been correctly saved in the PMS.


2

Test incoming Rate update

(OTA_HotelRatePlanNotifRQ)

  • Customer will have a rate code ready for the importing rates from the integration partner so the test could be in this rate code. if this rate code is not yet ready the integration partner should ask the customer to create a Test rate code
  • after the Test rate code has been created, ask the integration partner to push some rates
  • Make sure that the rate which needs to be updated is set to yieldable. Ask in the channel for which day the rate has been pushed, check the date and send a screenshot from the Protel Daily Rates if the rate for the specific date is showing in the list. 
  • as soon as this has been tested successfully repeat the tests for incoming rate updates for a rate or rate group chosen by the customer

3

Test incoming Restrictions 

(OTA_HotelAvailNotifRQ)

  • Make sure that the rate which needs to be updated is set to yieldable.
  • Ask the Integration Part in the Slack channel of the integration to push a restriction for a specific day on the Test rate code and then check in the Price availability for the given date if the restriction has been set.
  • Interalp supports only
    1. Close
    2. Close to Arrival
    3. Minimum Length of Stay By Arrival


4

Ask the Integration partner to clarify with the customer to have the Test rate code invalidated (through expired validity) after the verification of the Restriction message is done


5

Update the developer portal with the information about the installed customer
