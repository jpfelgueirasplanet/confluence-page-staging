Configure the Transaction Accounts in the System Data

  • all transaction accounts in protel need  to be mapped against a GHA DISCOVERY revenue type, otherwise the Stay Message will fail.

Recommendation:

To easier process the list of your TAA's, we recommend the following:
Go to protel System Data > Bookkeeping > Revenue and Payments > Transaction Accounts (TAA)

In the Transaction Account dialog, open the context menu by clicking right. Select either Export to Microsoft Excel or Export to HTML, to create an export of the list that you can use to do the mapping.

Tick "Show hotels" to see what hotels in a MPE are using the respective transaction account.

The hotel can use this list, to decide what revenue types the transaction accounts should be mapped against. Afterwards, the mapping can be proceeded as usual in the protel IDS (see below).

Configure the DISCOVERY Dollars in the System Data

Open the "Loyalty points" menu via Bookkeping > Revenue and Payments > Loyalty points


The window "Loyalty points" opens. This is where all information for using DISCOVERY Dollars will need to be entered.




Loyalty progams : GHA DISCOVERY




























Currency : D$


First, please configure the required currency "DISCOVERY Dollars" (short: D$). Create a new currency via the button "Open currencies".

Please make sure to configure the ISO 4271 in the dialog "Currencies" for all of you local currencies, otherwise protel SPE/MPE will not be able to provide this information in the messages to GRAVTY!


Payment methods : DISCOVERY Dollars

Afterwards, a new method of payment needs to be configured. To to so, click on "Open payments" in the Loyalty points dialog.

Important: the payment method needs to be configured with "Posting to A/R". This allows you to check the DISCOVERY Dollar revenue on a regular base, as well as cross check D$ reimbursements from GHA to the hotel.


Correction transaction account : 

The "Correction transaction account" serves as a transaction account for all differences that may occur through roundoff errors or fluctuations in exchange rates.

protel dll Setup

GHA also uses a special validation system for their loyalty cards. This is executed by means of the dll (Dynamic Link Library) settings. Depending on the operating system, the 64bit or the 32bit version is required.

Please follow the instructions to install the respective PROTEL dll Setup on every device working with protel.

You can download the dll-file from our protel Downloads page.

To install this dll, copy the file into the local protel folder, start the .exe, and follow the instructions.

Please select the local protel folder to install the dll.

This task should be given to the Hotel-IT as they know best how many stations there are.

After the installation, you will find the “LoyaltyCardsValidator.dll” in the folder C:\Windows\SysWOW64. This dll-file checks the validation of each GHA loyalty card. If a user then tries to enter an invalid loyalty card, an error message appears:

Configuring an IDS Instance for I/O: How to configure a protel I/O IDS

  • Token and module name

    • To add the token to your protel IDS instance, please go to the Common tab in the protel IDS settings. The token will be provided by our support team in Dortmund.

      The target is always “com.oracle.oxi”.

After you added the token it is visible in the Special tab.






  • Export URL

    • Add the following endpoint for the URLs for the export:

      • https://service.protel.io/services/ProtelApiService.ProtelApiServiceHttpsSoap12Endpoint







  • Import URL

    • Add the following URL as reservation import request for the import URL:

      • https://service.protel.io/services/ProtelApiService.ProtelApiServiceHttpsSoap12Endpoint






  • ESB server connection
    • Add the server URL. If it is a live customer, the URL is always “wss://wsesb.protel.io”. Add the URL in Menu [Global > ESB server connection options].









  • Data Export
    • [protel IDS settings > Special tab > Data export]
    • The result messages for the GHA DISCOVERY Interface are not always in-time. To prevent the IDS from going into an endless export loop the “fake” sync mode needs to be enabled. The Timeout in seconds needs to be 999. - ***Please note that newer versions of the IDS do not have the “fake” sync mode option any more.






  • Export availability values via IDS (OTA_HotelInvCountNotifRQ – RTAV)
    • Select the Enable calculation checkbox located in the Global available field. This transmits availabilities not only at room type level, but also at property level. After that, please make sure to disable the Export Availability Values check box.







  • Connections to a yield management system
    • If the hotel has a connection to a yield management system and the respective decisions should also be exported via the IDS, the following settings need to be made. In the IDS settings, go into the Export tab and choose the Yield Management subtab. Select the two checkboxes to activate the export.


    • Please note: Do not execute this setting if no yield management system has been connected! Doing so may result in the transfer of wrong restrictions!






  • Export rate values via IDS (OTA_HotelRatePlanNotifRQ – RATE)
    • Exclude 0-Rates from the Export
      • Open the protel IDS instance, go to the Export tab and choose the Rate Values subtab. Select the Misc. tab, tick the check box “Exclude rates from the export if all rates for that day are 0”. If this check box is not set, 0,00 rates will be sent for not configured rates in the future.
    • Export Child Rates
      • If child rates are to be exported as well, make sure the Children age ranges are properly configured and confirmed by the hotel. In protel IDS, configure the children rates in the Common tab. In the middle part of the screen, switch to the Children age ranges tab.
    • Send invalid rates to GHA
      • There are several scenarios in which an invalid rate code detail can be sent to GHA:
        • Validity of the rate has been reduced
        • Room type has been deletetd from a rate code detail
        • Room type in a rate code detail has been changed or modifyed
        • Rate code detail has been deleted

If one of these scenarios occurs, a respective message is sent. Furthermore, the Oxi-proxy message will be sent.

The Oxi-proxy is used to implement a mapping, because before an inactive rate code detail can be sent, an active one needs to be transmitted first.

To send a rate detail message, GHA requires a rate value for the respective time stamp. Unfortunately, the IDS is not able to provide a rate amount. Therefore a “default” rate amount needs to be set in BOB. This “default” rate amount needs to be discussed with the hotel.

In most cases, this rate amount will never appear on the GHA website, since the message containing the inactive rate detail will follow the next second. But if for any reason the inactive message fails, the „default“ rate amount is required. Please suggest to the hotel to use a high „default“ rate amount.

Directly after a time period has been activated, the inactive message will be sent to GHA. The identification code for an inactive rate detail is the following:

RateHeader / RateDetails / @mfInactiveDate.

The current date and timestamp will be used.


  • Settings in the IDS

    • The functionality required to send invalid rates is included in every version of the IDS, starting with version 5.322, so please make sure this version is at least installed.


      As soon as this setting is made, the following values will be part of the OTA_HotelRatePlanNotifRQ message:

      • RateTier= “Active”
      • RateTier= “Deactivated”

      These elements are required to send inactive rates to GHA.







  • Import Reservation

    • Import Notes

      • As GHA provides the user with comments per reservation, the customer can decide where they should be saved after the import. The possibilities are Note1, Note2 or Memo.

        Please configure them in the IDS settings in the Text processing subtab located under the Import tab as follows:

      • Please note: By “import and export” with regards to reservations, it means that the data is “requested” by the protel PMS and “fetched” from the GHA server.

    • Import Children

      • It is also possible to import reservations with children included. The IDS uses the same age ranges for children as defined for the rate exports.

        In the incoming messages, the following tag is included:

        <GuestCount AgeQualifyingCode="8" Count="1" Age="7"/>

        In this example, “Child 2” (age 6-11) is going to be imported.

        No additional setting is needed, if the child age ranges are enable for the export.

Required IDS Mapping

The required IDS Mapping can also be done by the customer. The respective mapping information can also be found in the Setup Guide in the protel SPE/MPE Online Help

If not the following Types needs to be mapped, please use the GHA Workbook for it


  • Map mpeHotel with the CloudID.
  • Map all room types.
  • All room types need to be mapped, since GHA DISCOVERY wants to display the same availability on the website as displayed in the protel PMS. GHA will never overbook the availability of a property, based on the availability of a property configured in the OCM.
  • Map all rates that should be exported to GHA DISCOVERY.
  • Use “FIX RATE“ for virtual rates; otherwise protel will charge supplements or discounts, since the calculated value has already been exported to GHA DISCOVERY. Furthermore, all mappings need to be entered without blanks – otherwise GHA DISCOVERY  will not be able to process the mappings!
  • Use the Type "market code" to map a MarketCode against a Rate Code. Of course use the GHA DISCOVERY market code value. The format needs to look like this RateCode#MarketCode (Since the structure of the rates differs between GHA DISCOVERY and protel, it is not possible to automatically transfer the assigned market code from the rate in the protel PMS. A rate can contain different market codes in protel and GHA DISCOVERY does not allow a rate to have several assigned market codes.)
  • Additionally to the RateCode#MarketCode, please create the normal MarketCode Mapping
  • Map all split tables to be exported from the rate code – the value must be equal to the split table’s name and should not contain spaces
  • The “Daily Rates“ split table does not have to be mapped, because the daily rates in the rate are calculated automatically.
  • If the split is mapped, please select “=” (dailyrate) as the value.
  • Map Gender types
  • Map Loyalty cards with member cards
  • Map Loyalty Levels for the loyalty cards
  • May Payment methods (find the GHA DISCOVERY values in the drop down box below)
  • Map Transaction Accounts
