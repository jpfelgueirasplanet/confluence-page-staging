Request authentication details from the Integration partner for the new installation and request them to send you the authentication details on a private Slack message or encrypted file by email and inform you by phone about the password.

Special XSetup Configuration

Set Default Cash Register for incoming PostPayments

This should be clarified in advance with the customer in case that there has to be a new, specific cash register to be set up.

In order to register the incoming PostPayments with a specific cash register number in pAir, the below configuration is needed to define and map a default cash register to tag on the incoming post payments through the HTNG_ChargePostingRQ.

Define a default cash register in pAir
System data / POS / Payment methods / POS

2. Add an xsetup that is used to define the default cash register for all IO-related postings in the configuration editor

  • Add CashRegister and default folder in Configuration/currentProperty/propData (Notice: "default" - low case letters !!!)
  • Add protelIO value with Content type: int and set the default cash register number in Data text box


Activate Batch Sending
(sum all items of a package into a Batch in the Folio message)

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelFolioRS/BuildBatchTree

  • Create HTNG_HotelFolioRS/BuildBatchTree folder if they do not exist
  • The Content type for HTNG_HotelFolioRS should be null
  • The content type for BuildBatchTree should be Boolean and set to "Yes"


Configure a KSK template for invoices

Configuration/currentProperty/propData/Interfaces/IO/Gateway/HTNG_HotelCheckOutNotifRS/TemplateNameInvoice

  • Create HTNG_HotelCheckOutNotifRS/TemplateNameInvoice folder if they do not exist
  • The Content type for HTNG_HotelCheckOutNotifRS is null
  • The content type for TemplateNameInvoice is String
  • The Data is the Name of the Template. Important: case sensitive!

  • The Invoice Template needs to be configured in the System Language

One way Installation

1

Create a test reservation in the future (at least one year after the system date) & provide the reservation number to your integration partner

  • make a modification and finally a cancellation of your test reservation. Verify in Kibana that outgoing OTA_HotelResNotifRQ is answered by OTA_HotelResNotifRS and no error occurs. 

2

Create and start bulk uploads

  • for reservations (OTA_HotelResNotifRQ)
    starting on the installation date and ending with 2099

BTS Jobs will be processed one by one as only one bulk per property is allowed at the same time. 

Bulks should be starting with 1 January and ending with 31 of December (i.e. 01.01.2017 - 31.12.2021).

In case you have issues with a stuck bulk upload, investigate why messages were not sent. As an option, delete the old job and create a new one. 

 

Two way Installation

Examples

1

Note:

  • the profile modifications and adding trace to reservations are done during the check-in step
  •  Hotelbird has pre-checkin status in their database.. this means that the guest is checked in at Hotelbird. This can already happen days/weeks before the day of the actual arrival...
    On the day of arrival Hotelbird will try to do the "real checkin" in the PMS. If the CheckIn fails (room not clean etc.) They will retry every 10 minutes.
  • Check-in is only possible for reservations with an assigned room


2

Test an incoming Check-in / Profile modification / Trace creation
(HTNG_HotelCheckInNotifRQ / OTA_ProfileModifyRQ / IO_ResModifyNotifRQ)

  • Create a test reservation (arrival date needs to be current PMS date) and provide the reservation number to your integration partner
  • Check whether the room has the correct status "clean" or "checked"
  • Ask the integration partner to check-in the test reservation
  • Verify in Kibana that the OTA_ProfileModifyRQ, IO_ResModifyNotifRQ and HTNG_HotelCheckInNotifRQ are answered by RS and no errors occurs
  • Check in the PMS if the profile (name, address, email...) was changed, the traces were added and the reservation was check-in successfully
  • Let the integration partner knows that you will cancel the reservation. Undo the check-in and cancel the test reservation afterwards

3

Test an incoming invoice detail request
(HTNG_HotelFolioRQ)

Note: Hotelbird can only handle 2 Folios. The First folio is always private and second always business.

  • Provide the reservation number of the test reservation prepared by the Hotel during the general preparation step
  • Ask the integration partner to view the invoice
  • Verify in Kibana that incoming HTNG_HotelFolioRQ is answered by HTNG_HotelFolioRS and contains no errors.
  • Open the invoice in the PMS and compare the amount of the invoice with the amount of the integration partner


4

Test an incoming payment posting
(HTNG_ChargePostingRQ)

  • Request the partner to pay the invoice of the previous reservation
  • The partner should use the PMS payment codes you provided during the general preparation step
  • Check if the amount was posted correctly in the PMS
  • Verify in Kibana that incoming HTNG_ChargePostingRQ is answered by HTNG_ChargePostingRS and no error occurs


5

Test an incoming check-out
(HTNG_HotelCheckOutNotifRQ)

  • Request the partner to check-out the reservation
  • Check in the PMS if the Reservation has been check-out
  • Verify whether the partner receives an email with the correct invoice
  • Verify in Kibana that incoming HTNG_HotelCheckOutNotifRQ is answered by HTNG_HotelCheckOutNotifRS and contains no errors.


6

Update the protel SMP page with the information about the installed customer
